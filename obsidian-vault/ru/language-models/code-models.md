# Языковые модели для кода

## Необходимые знания

[[llm]]

## Что это такое

Языковая модель для кода — это языковая модель, обученная преимущественно или в значительной мере на исходном коде, с целью понимания и генерации программ, а не (или в дополнение к) текстовой прозе. Определяющее свойство — не отдельная архитектура, а сочетание состава обучающих данных, специализированных целей предобучения и методологии оценки, ориентированных на формальную, исполняемую, многоязыковую природу кода.

Код занимает уникальную позицию среди доменов языкового моделирования. Он формально структурирован (синтаксис однозначен, в отличие от естественного языка), исполняем (выходы можно механически верифицировать запуском), многоязычен в богатом смысле (десятки языков программирования с разной семантикой сосуществуют в обучающих данных) и самодокументирован (комментарии и строки документации в тех же файлах, что и код, создают естественное выравнивание между естественным языком и формальными программами).

## Архитектура

Модели кода используют ту же декодерную трансформерную архитектуру, что и общие LLM. Отличия — в стратегии токенизации и целях предобучения.

**Токенизация**: стандартные BPE-токенайзеры, обученные на коде, производят более детальные токены, чем текстовые токенайзеры — идентификаторы, ключевые слова и операторы часто являются отдельными токенами. Отступы представляют значительную проблему для Python и YAML, где пробельные символы семантически значимы.

**Длина контекста**: понимание кода на уровне репозитория требует длинного контекста. Вызов функции может быть отделён от её определения тысячами строк. Современные модели кода поддерживают контексты от 8K до 128K токенов.

**Позиционное кодирование**: RoPE с настройкой базовой частоты для экстраполяции на длинные контексты, как в общих LLM.

## Математический аппарат

**Стандартное предсказание следующего токена** на корпусах кода:

$$\mathcal{L}_{\text{CLM}} = -\sum_{t=1}^{T} \log p_\theta(x_t \mid x_{<t})$$

**Заполнение середины (Fill-in-the-Middle, FIM)** — преобразование обучения, учащее модель предсказывать фрагмент кода по заданным префиксу и суффиксу. Документ $(P, M, S)$ (префикс, середина, суффикс) переставляется в формат $\langle\text{PRE}\rangle P \langle\text{SUF}\rangle S \langle\text{MID}\rangle M$ и модель предсказывает $M$ авторегрессионно:

$$\mathcal{L}_{\text{FIM}} = -\sum_{t=1}^{|M|} \log p_\theta(m_t \mid \langle\text{PRE}\rangle, P, \langle\text{SUF}\rangle, S, \langle\text{MID}\rangle, m_{<t})$$

FIM принципиально важен для интеграции в IDE: автодополнение должно заполнять пробел в середине файла, а не только дописывать в конец.

**Метрика HumanEval pass@k** — измеряет вероятность того, что хотя бы одна из $k$ сгенерированных выборок пройдёт все модульные тесты. Оценивается без генерации $\binom{n}{k}$ комбинаций, выборкой $n > k$ решений и подсчётом $c$ прошедших:

$$\text{pass@}k = 1 - \frac{\binom{n - c}{k}}{\binom{n}{k}}$$

Например, при $n = 10$ выборках и $c = 4$ прошедших $\text{pass@}1 = 1 - \binom{6}{1}/\binom{10}{1} = 0{,}4$.

**Целевая функция с обратной связью от выполнения (RLEF)** — дообучение с сигналом награды от запуска тестов. Модель генерирует решения; каждое исполняется; результат прохождения/провала теста становится сигналом награды в обновлении PPO или REINFORCE.

## Парадигма обучения

**Состав данных**: обучение моделей кода смешивает репозитории GitHub (отфильтрованные по лицензии и качеству), документацию по программированию, пары вопрос-ответ Stack Overflow, задачи спортивного программирования с решениями и часто долю текста на естественном языке для сохранения следования инструкциям.

**Дедупликация**: код на GitHub высоко дублирован (форки репозиториев делят идентичные файлы). Удаление почти-дубликатов с помощью MinHash LSH необходимо: обучение на дублированном коде завышает результаты бенчмарков без улучшения реального обобщения.

**Многоэтапное обучение**:
1. Общее предобучение на смешанном корпусе код + текст.
2. Продолженное предобучение с акцентом на код.
3. Обучение FIM — либо встроенное в предобучение, либо как отдельный этап.
4. Дообучение с учителем (SFT) на парах следования инструкциям.
5. RLEF — дообучение с обратной связью от выполнения.

## Ключевые свойства и возможности

- **Автодополнение кода**: предсказание следующего токена и FIM для завершения частичного кода в редакторе.
- **Генерация документации**: создание объяснений на естественном языке из кода.
- **Перевод между языками**: модели, обученные на нескольких языках программирования, учатся общим абстракциям и могут транслировать код.
- **Обнаружение и исправление ошибок**: модель, дообученная на парах (баг, исправление), может находить и чинить ошибки.
- **Генерация тестов**: генерация модульных тестов из реализаций функций.
- **Понимание репозитория**: при длинном контексте модели могут понимать граф вызовов, зависимости модулей и иерархии классов.

## Сравнение с другими типами

| Параметр | Модель кода | Общая LLM | SLM |
|---|---|---|---|
| Основная сила | Синтез кода, отладка | Широкие NLU/NLG | Вывод на устройствах |
| Метрика оценки | pass@k (исполнение) | Перплексия, MMLU | Эффективность + качество |
| Данные обучения | Код-dominant | Веб-текст | Отобранный текст |
| Поддержка FIM | Да | Редко | Иногда |
| Применимость RLEF | Да | Ограничена | Ограничена |
| Потребность в контексте | Длинная (репозиторий) | Переменная | Короткая |

## Пример использования на Python

```python
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_id = "bigcode/starcoder2-7b"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# Стандартное завершение (каузальное, слева направо)
prompt = "def binary_search(arr: list[int], target: int) -> int:\n    "
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
with torch.no_grad():
    output = model.generate(
        **inputs,
        max_new_tokens=256,
        temperature=0.2,
        do_sample=True,
        pad_token_id=tokenizer.eos_token_id,
    )
print(tokenizer.decode(output[0], skip_special_tokens=True))

# FIM (заполнение середины)
prefix = "def compute_entropy(probs: list[float]) -> float:\n    "
suffix = "\n    return entropy"
fim_prompt = f"<fim_prefix>{prefix}<fim_suffix>{suffix}<fim_middle>"
fim_inputs = tokenizer(fim_prompt, return_tensors="pt").to(model.device)
with torch.no_grad():
    fim_output = model.generate(
        **fim_inputs,
        max_new_tokens=128,
        temperature=0.1,
        do_sample=True,
    )
middle = tokenizer.decode(fim_output[0], skip_special_tokens=True)
print(f"{prefix}{middle}{suffix}")
```

## Ограничения

- **Корректность против правдоподобия**: модели генерируют синтаксически правдоподобный код, который часто содержит тонкие семантические ошибки — ошибки на единицу, неправильное использование API, пропуск граничных случаев.
- **Переобучение на бенчмарках**: HumanEval и MBPP относительно малы и могут присутствовать в обучающих данных. SWE-Bench (реальные задачи GitHub) сложнее переобучить.
- **Редкие языки и фреймворки**: обучающие данные доминированы Python, JavaScript и Java. На Rust, Zig, Fortran модели работают значительно хуже.
- **Дальние зависимости**: даже при контексте 128K отслеживание сложных кросс-файловых зависимостей может превышать эффективный бюджет внимания.
- **Безопасность**: модели могут генерировать небезопасный код (SQL-инъекции, переполнение буфера) без явных ограничений.
- **Незнание среды выполнения**: модель не имеет доступа к среде выполнения во время генерации, что приводит к галлюцинированным импортам и несуществующим API.

## Связанные темы

[[llm]] — модели кода являются специализированными LLM с той же фундаментальной архитектурой  
[[slm]] — небольшие модели кода (1–3B параметров) эффективны для дополнения одиночных функций на устройстве  
[[mixture-of-experts]] — MoE-модели кода могут направлять токены к специализированным по языку экспертам  
[[pac-learning]] — формальный теоретико-обучающий анализ условий обобщения синтеза кода
