# Смесь экспертов (MoE)

## Необходимые знания

[[llm]], [[slm]]

## Что это такое

Смесь экспертов (Mixture of Experts, MoE) — архитектурная парадигма, разделяющая число параметров модели и объём вычислений на один прямой проход. Стандартный плотный трансформер использует каждый параметр для каждого токена. MoE-трансформер заменяет некоторые или все слои прямого распространения банком из $N$ экспертных сетей и обученным маршрутизатором, который выбирает малое подмножество $k$ из них для каждого токена. Только выбранные $k$ экспертов выполняют вычисления; остальные простаивают для данного токена.

Следствие разительное: можно иметь модель с, допустим, 200 миллиардами параметров, но использовать вычисления, эквивалентные плотной модели в 20 миллиардов, поскольку активна лишь доля $k/N$ экспертов. Это даёт MoE выгодное положение на кривой качество-вычисления: тот же бюджет FLOP, что у плотной модели, но доступ к гораздо большему числу параметров и, следовательно, большей репрезентативной ёмкости.

MoE — не новая идея: Jacobs et al. ввели её в 1991 году, — но она стала практичной для крупномасштабного языкового моделирования со Switch Transformer (2021) и с тех пор стала доминирующей архитектурой во фронтирных моделях.

## Архитектура

В трансформере MoE каждый стандартный FFN-слой заменяется **MoE-слоем**, состоящим из:

1. **$N$ экспертных FFN**: каждый эксперт — независимый двухслойный MLP, но инициализированные по-разному и учащиеся специализироваться.
2. **Маршрутизатор (гейтинговая сеть)**: линейный слой $W_g \in \mathbb{R}^{d \times N}$ с последующим softmax, производящий распределение вероятностей по экспертам для каждого токена.

Маршрутизация и выбор экспертов:

$$g(x) = \text{Softmax}\!\left(\text{TopK}\!\left(x W_g,\; k\right)\right)$$

где $\text{TopK}(z, k)$ сохраняет топ-$k$ логитов, остальные устанавливает в $-\infty$ перед softmax, чтобы только $k$ экспертов получали ненулевой вес. Выход MoE-слоя — взвешенная сумма выбранных экспертов:

$$\text{MoE}(x) = \sum_{i \in \text{TopK}} g(x)_i \cdot \text{FFN}_i(x)$$

Обычно $k = 1$ (Switch Transformer) или $k = 2$ (GShard, Mixtral).

## Математический аппарат

**Логиты маршрутизатора и гейтинг**:

$$h(x) = x W_g \in \mathbb{R}^N$$
$$g(x) = \text{Softmax}(\text{TopK}(h(x), k))$$

**Вспомогательная потеря балансировки нагрузки** — ключевой механизм стабильности. Без неё маршрутизатор коллапсирует к постоянному выбору одного-двух экспертов, оставляя остальных недообученными. Вспомогательная потеря штрафует за неравномерную утилизацию экспертов:

$$\mathcal{L}_{\text{aux}} = \alpha \cdot N \cdot \sum_{i=1}^{N} f_i \cdot P_i$$

где $f_i$ — доля токенов, направленных к эксперту $i$ в батче (фактическая нагрузка), $P_i$ — средняя вероятность маршрутизации, назначенная эксперту $i$ по батчу (дифференцируемый суррогат), $\alpha$ — малый коэффициент (обычно $10^{-2}$–$10^{-4}$). Произведение $f_i P_i$ дифференцируемо по параметрам маршрутизатора через $P_i$.

**Коэффициент ёмкости** — каждый эксперт имеет максимальную ёмкость по токенам $C = \text{CapacityFactor} \times \frac{\text{tokens\_per\_batch}}{N}$. Токены, направленные к переполненному эксперту, отбрасываются. Значение 1,25 типично.

## Парадигма обучения

Обучение MoE — стандартное авторегрессионное предсказание следующего токена плюс вспомогательная потеря балансировки:

$$\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{LM}} + \mathcal{L}_{\text{aux}}$$

Ключевые трудности:

- **Инициализация маршрутизатора**: маршрутизатор не должен коллапсировать в начале обучения. Равномерная инициализация или малый шум в $W_g$ помогает.
- **Экспертный дропаут**: случайное выключение выходов экспертов при обучении служит регуляризацией.
- **Параллелизм экспертов**: при распределённом обучении разные эксперты размещаются на разных устройствах. Операция «все-ко-всем» для диспетчеризации токенов — главные накладные расходы.
- **Поток градиентов**: градиенты проходят через веса гейтинга только для топ-$k$ выбранных экспертов, создавая разреженность в графе градиентов.

## Ключевые свойства и возможности

- **Эффективность параметров**: то же качество, что у плотной модели, при меньших вычислениях на токен, или лучшее качество при том же вычислительном бюджете.
- **Специализация экспертов**: эксперты учатся обрабатывать разные подмножества входного распределения — разные языки, синтаксические структуры или тематические домены, — хотя эта специализация частична и эмерджентна.
- **Масштабирование**: MoE масштабирует эффективную ёмкость модели без пропорционального масштабирования FLOP.
- **Требования к памяти**: все $N$ экспертов должны находиться в памяти, даже если только $k$ активны — основная сложность развёртывания.
- **Отбрасывание токенов**: токены, направленные к переполненным экспертам, молча отбрасываются, что может незаметно снижать точность.

## Сравнение с другими типами

| Параметр | Плотная LLM | MoE LLM | SLM |
|---|---|---|---|
| Суммарные параметры | N | N × эксперты | Малые |
| Активные параметры на токен | N | N / эксперты × k | Малые |
| FLOP на токен | Высокие | Низкие | Низкие |
| Объём памяти | N | N × эксперты | Малый |
| Стабильность обучения | Высокая | Умеренная | Высокая |
| Баланс нагрузки | Н/П | Требует настройки | Н/П |

## Пример использования на Python

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class TopKRouter(nn.Module):
    def __init__(self, d_model: int, num_experts: int, top_k: int):
        super().__init__()
        self.gate = nn.Linear(d_model, num_experts, bias=False)
        self.num_experts = num_experts
        self.top_k = top_k

    def forward(self, x: torch.Tensor):
        logits = self.gate(x)
        top_k_logits, top_k_indices = logits.topk(self.top_k, dim=-1)
        zeros = torch.full_like(logits, float("-inf"))
        sparse_logits = zeros.scatter(-1, top_k_indices, top_k_logits)
        router_probs = F.softmax(sparse_logits, dim=-1)

        # Вспомогательная потеря балансировки нагрузки
        f = (top_k_indices.unsqueeze(-1) == torch.arange(
            self.num_experts, device=x.device
        )).float().mean(dim=(0, 1))
        p = F.softmax(logits, dim=-1).mean(dim=(0, 1))
        aux_loss = self.num_experts * (f * p).sum()

        return router_probs, top_k_indices, aux_loss


class MoELayer(nn.Module):
    def __init__(self, d_model: int, d_ff: int, num_experts: int, top_k: int = 2):
        super().__init__()
        self.router = TopKRouter(d_model, num_experts, top_k)
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(d_model, d_ff),
                nn.GELU(),
                nn.Linear(d_ff, d_model),
            )
            for _ in range(num_experts)
        ])
        self.top_k = top_k

    def forward(self, x: torch.Tensor):
        B, T, D = x.shape
        x_flat = x.view(B * T, D)
        router_probs, indices, aux_loss = self.router(x_flat.unsqueeze(0))
        router_probs = router_probs.squeeze(0)
        indices = indices.squeeze(0)

        output = torch.zeros_like(x_flat)
        for k in range(self.top_k):
            expert_idx = indices[:, k]
            for i, expert in enumerate(self.experts):
                mask = (expert_idx == i)
                if mask.any():
                    output[mask] += (
                        router_probs[mask, expert_idx[mask]].unsqueeze(-1)
                        * expert(x_flat[mask])
                    )
        return output.view(B, T, D), aux_loss
```

## Ограничения

- **Асимметрия памяти и вычислений**: полный банк экспертов должен помещаться в память устройства, хотя каждый токен использует только $k/N$ из него. Обслуживание модели 200B MoE требует инфраструктуры для 200B параметров, даже если каждый токен потребляет ~20B-эквивалент FLOP.
- **Нестабильность маршрутизации**: в начале обучения маршрутизатор может коллапсировать на несколько экспертов. Восстановление из коллапса медленное.
- **Накладные расходы на коммуникацию**: параллелизм экспертов требует операций «все-ко-всем» для диспетчеризации токенов между устройствами.
- **Отбрасывание токенов и несогласованность**: отброшенные токены получают нулевую обработку от пропущенного эксперта.
- **Сложность дообучения**: дообучение MoE-моделей на малых наборах данных труднее, чем для плотных моделей.
