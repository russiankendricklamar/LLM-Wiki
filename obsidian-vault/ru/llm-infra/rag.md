# RAG (Retrieval-Augmented Generation)

## Обзор

Retrieval-Augmented Generation (RAG) — архитектурный паттерн, который повышает качество ответов языковых моделей за счёт подачи релевантных внешних документов непосредственно во время инференса. Вместо того чтобы полагаться исключительно на знания, закодированные в параметрах при предобучении, RAG-система сначала извлекает небольшое число релевантных фрагментов из корпуса, а затем передаёт их модели вместе с исходным запросом.

Мотивация очевидна: языковые модели имеют фиксированную дату отсечения обучающих данных и ограниченное контекстное окно. Они галлюцинируют факты, когда речь заходит о корпоративных данных, недавних событиях или редких темах. RAG решает обе проблемы без дорогостоящего тонкого дообучения. Подход был представлен в работе Lewis et al. (Facebook AI Research, 2020) «Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks» и с тех пор стал доминирующим паттерном заземления ответов LLM на проверяемые источники.

С производственной точки зрения RAG разделяет базу знаний и веса модели. Корпус для поиска можно обновлять, не трогая модель. Эта независимость особенно привлекательна для корпоративных развёртываний, где документы регулярно меняются и требования к соответствию предполагают трассируемые ссылки на источники.

## Как это работает

RAG-пайплайн состоит из двух фаз: **офлайн-индексация** и **онлайн-поиск + генерация**.

**Офлайн-индексация:**
1. Исходные документы (PDF, HTML, текстовые файлы) разбиваются на фрагменты (чанки).
2. Каждый фрагмент кодируется энкодером (например, `text-embedding-3-small`, `all-MiniLM-L6-v2`) в плотный вектор в $\mathbb{R}^d$.
3. Векторы и соответствующий текст сохраняются в векторной базе данных (FAISS, Pinecone, Weaviate, Chroma).

**Онлайн-поиск + генерация:**
1. Запрос пользователя кодируется тем же энкодером.
2. Ближайшие $k$ фрагментов извлекаются алгоритмом приближённого поиска ближайших соседей (ANN).
3. Найденные фрагменты конкатенируются в контекст промпта.
4. Языковая модель генерирует ответ, обусловленный запросом и найденным контекстом.

Стратегия разбиения на чанки — критически важное инженерное решение. Разбиение фиксированного размера (например, 512 токенов с перекрытием 64) простое и хорошо работает для однородных документов. Семантическое разбиение делит на абзацы или секции, обнаруживаемые небольшим классификатором предложений. Разбиение по предложениям с окном вокруг них улучшает баланс точности и полноты.

## Математическое описание

Пусть $q$ — запрос, $\mathcal{D} = \{d_1, \ldots, d_N\}$ — корпус документов. RAG определяет совместную вероятность над выходной последовательностью $y$:

$$p(y \mid q) = \sum_{d \in \text{top-}k(\mathcal{D}, q)} p(d \mid q) \cdot p(y \mid q, d)$$

На практике маргинализация аппроксимируется использованием только top-$k$ найденных документов. Оценка релевантности $p(d \mid q)$ в плотном поиске:

$$\text{score}(q, d) = \frac{\mathbf{e}_q \cdot \mathbf{e}_d}{\|\mathbf{e}_q\| \cdot \|\mathbf{e}_d\|}$$

где $\mathbf{e}_q, \mathbf{e}_d \in \mathbb{R}^d$ — эмбеддинги двухоэнкодерной модели.

Для разреженного поиска BM25 оценивает документ для запроса $q = \{t_1, \ldots, t_m\}$ как:

$$\text{BM25}(q, d) = \sum_{i=1}^{m} \text{IDF}(t_i) \cdot \frac{f(t_i, d) \cdot (k_1 + 1)}{f(t_i, d) + k_1 \cdot \left(1 - b + b \cdot \frac{|d|}{\text{avgdl}}\right)}$$

Гибридный поиск линейно комбинирует плотный и разреженный сигналы с весом $\alpha$:

$$\text{score}_{\text{hybrid}} = \alpha \cdot \text{score}_{\text{dense}} + (1 - \alpha) \cdot \text{score}_{\text{sparse}}$$

## Реализация

```python
from sentence_transformers import SentenceTransformer
import faiss
import numpy as np
from anthropic import Anthropic


def chunk_text(text: str, chunk_size: int = 400, overlap: int = 50) -> list[str]:
    words = text.split()
    chunks = []
    start = 0
    while start < len(words):
        end = min(start + chunk_size, len(words))
        chunks.append(" ".join(words[start:end]))
        start += chunk_size - overlap
    return chunks


def build_index(documents: list[str], model_name: str = "all-MiniLM-L6-v2"):
    model = SentenceTransformer(model_name)
    chunks: list[str] = []
    for doc in documents:
        chunks.extend(chunk_text(doc))

    embeddings = model.encode(chunks, convert_to_numpy=True, normalize_embeddings=True)
    dim = embeddings.shape[1]

    # IndexFlatIP = inner product, что равно косинусному сходству на нормализованных векторах
    index = faiss.IndexFlatIP(dim)
    index.add(embeddings.astype(np.float32))
    return index, chunks, model


def retrieve(query: str, index, chunks: list[str], model, k: int = 4) -> list[str]:
    q_emb = model.encode([query], convert_to_numpy=True, normalize_embeddings=True)
    _distances, indices = index.search(q_emb.astype(np.float32), k)
    return [chunks[i] for i in indices[0]]


def rag_answer(query: str, retrieved: list[str], client: Anthropic) -> str:
    context = "\n\n---\n\n".join(retrieved)
    prompt = (
        f"Используй только приведённый контекст для ответа на вопрос.\n\n"
        f"Контекст:\n{context}\n\nВопрос: {query}"
    )
    message = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}],
    )
    return message.content[0].text


# Пример использования
documents = [
    "Архитектура трансформера была представлена в статье 'Attention Is All You Need' (Vaswani et al., 2017).",
    "FAISS — библиотека для эффективного поиска сходства, разработанная Facebook AI Research.",
    "BM25 — вероятностная модель ранжирования, расширяющая TF-IDF нормализацией по длине документа.",
]

index, chunks, embed_model = build_index(documents)
client = Anthropic()
query = "Что такое FAISS?"
retrieved_chunks = retrieve(query, index, chunks, embed_model)
answer = rag_answer(query, retrieved_chunks, client)
print(answer)
```

```
                   │
                   ▼

                   │
                   ▼

```

## Ключевые компромиссы

- **Решает:** устаревание знаний модели, галлюцинации по фактическим вопросам, ограниченное контекстное окно для больших корпусов, требования к цитированию источников.
- **Не решает:** ошибки рассуждений, многошаговый вывод по многим документам без ранжирования, атакующие документы в корпусе.
- **Когда использовать:** корпоративные или часто обновляемые базы знаний, требования комплаенса к цитированию, когда бюджет на дообучение недоступен.
- **Когда не использовать:** задачи, требующие глубокой стилистической адаптации (лучше дообучение), высококритичные по задержке пайплайны.
- **Размер чанков:** большие чанки улучшают полноту, маленькие — точность. Перекрытие предотвращает разрыв релевантной информации по границам чанков.

## Варианты и расширения

**Гибридный поиск** объединяет плотный и разреженный сигналы. Алгоритм Reciprocal Rank Fusion (RRF) сливает ранжированные списки без нормализации оценок.

**Переранжирование** применяет кросс-энкодер (например, `cross-encoder/ms-marco-MiniLM-L-6-v2`) как второй проход для переранжирования top-$n$ плотных результатов. Кросс-энкодеры видят запрос и документ одновременно, обеспечивая более богатое взаимодействие ценой задержки.

**HyDE (гипотетические эмбеддинги документов)** — языковая модель генерирует гипотетический ответ на запрос, его эмбеддинг используется для поиска реальных документов. Это выравнивает пространство эмбеддингов запросов и документов.

**Агентный RAG** использует [[tool-use]], превращая поиск в инструмент, который модель может вызывать итеративно, комбинируя несколько запросов с рассуждением через [[chain-of-thought]].

## Оценка качества

Фреймворк RAGAS измеряет:
- **Достоверность (Faithfulness):** содержит ли ответ только утверждения, подкреплённые найденным контекстом?
- **Релевантность ответа:** семантически ли согласован ответ с вопросом?
- **Полнота контекста (Context recall):** содержит ли найденный контекст всю необходимую информацию?
- **Точность контекста (Context precision):** какая доля найденного контекста действительно релевантна?

Низкая достоверность указывает, что LLM игнорирует найденный контекст. Низкая полнота контекста указывает на сбои поиска.

## Практические применения

- **Корпоративные Q&A по внутренним документам:** юридические, HR, инженерные базы знаний.
- **Поддержка клиентов:** руководства по продуктам, инструкции по устранению неполадок.
- **Поиск по коду и объяснение:** кодовая база как корпус, запросы о функциях и API.
- **Финансовые исследования:** отчёты о доходах, регуляторные документы, рыночные данные.
- **Медицина:** клинические рекомендации, базы лекарственных взаимодействий.
