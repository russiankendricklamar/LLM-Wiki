# Вызов инструментов / Function Calling

## Обзор

Вызов инструментов (tool use, function calling) — возможность языковых моделей вызывать внешние функции, API и сервисы структурированным, программным способом. Вместо того чтобы возвращать неструктурированный текст с описанием того, что следовало бы сделать, модель с поддержкой инструментов формирует машиночитаемый вызов: имя инструмента и типизированный объект аргументов. Вызывающий код исполняет инструмент, возвращает результаты модели, и генерация продолжается.

Эта возможность превращает языковые модели из обработчиков текста в агентов, способных читать живые данные, записывать файлы, запускать код и взаимодействовать с внешними системами. Впервые она получила широкое распространение в продакшне после релиза function calling от OpenAI в июне 2023 года, за которым последовали Anthropic, Google и другие. Механизм прост: модель обучается распознавать, когда требуется внешняя информация или действие, и выражать эту потребность в структурированном JSON-формате, который слой приложения может разобрать и направить куда нужно.

Вызов инструментов лежит в основе автономных агентов, copilot-систем и любых решений, где [[llm|LLM]] должна действовать в реальном мире, а не просто описывать его. В сочетании с [[rag]] инструментально оснащённые модели могут решать, когда искать, а когда отвечать из обучающих данных. В сочетании с рассуждением через [[chain-of-thought]] они реализуют паттерн ReAct (Reasoning + Acting).

## Как это работает

**Шаг 1 — Определение схемы инструментов.** Разработчик предоставляет JSON Schema, описывающую доступные инструменты: имя, описание, типы параметров. Эта схема внедряется в системный промпт или специальное поле `tools` в API-запросе.

**Шаг 2 — Решение модели.** Модель решает, отвечать напрямую или вызвать инструмент. Это решение неявное: вместо обычного текста модель генерирует блок `tool_use` (Anthropic) или объект `function_call` (OpenAI), когда определяет необходимость внешней информации.

**Шаг 3 — Выполнение инструмента.** Прикладной уровень разбирает структурированный вывод, валидирует входные данные, выполняет функцию и получает результаты.

**Шаг 4 — Внедрение результата.** Результаты инструмента добавляются в беседу как сообщение `tool_result`. Модель получает их и продолжает генерацию.

**Шаг 5 — Завершение.** Цикл повторяется, пока модель не сгенерирует ответ без вызовов инструментов, сигнализируя о достаточности информации.

Многоходовый вызов инструментов позволяет цепочки: в одном агентном цикле модель может вызвать `search_web`, затем `summarize_document`, затем `write_email`, каждый раз опираясь на предыдущий шаг.

Параллельные вызовы инструментов (поддерживаются Anthropic и OpenAI) позволяют модели одновременно выдавать несколько вызовов, когда они независимы, снижая задержку за счёт параллельного выполнения.

## Математическое описание

Фреймворк ReAct формализует вызов инструментов как чередование рассуждений и действий. Агент производит последовательность троек:

$$\tau = [(r_1, a_1, o_1), (r_2, a_2, o_2), \ldots, (r_T, a_T, o_T)]$$

где $r_t$ — трасса рассуждений (цепочка мысли), $a_t$ — действие (вызов инструмента), $o_t$ — наблюдение (результат инструмента). Политика $\pi_\theta$ параметризует модель:

$$a_t = \pi_\theta(q, \tau_{<t})$$

Выбор инструмента — это классификация по множеству инструментов $\mathcal{T}$. Для истории беседы $h_t$ модель присваивает вероятность:

$$p(a_t = \text{call}(f, \mathbf{x}) \mid h_t) \propto \exp(\text{совместимость}(f, h_t))$$

Для обеспечения безопасности входные аргументы $\mathbf{x}$ должны удовлетворять ограничениям JSON Schema перед выполнением:

$$\text{valid}(\mathbf{x}) \iff \mathbf{x} \models \text{schema}(f)$$

## Реализация

```python
import anthropic
import json

client = anthropic.Anthropic()

# Определения инструментов (JSON Schema)
tools = [
    {
        "name": "get_stock_price",
        "description": "Получить текущую цену акции по тикеру.",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {
                    "type": "string",
                    "description": "Тикер акции, например AAPL",
                },
            },
            "required": ["ticker"],
        },
    },
    {
        "name": "calculate_portfolio_value",
        "description": "Вычислить общую стоимость портфеля по позициям и ценам.",
        "input_schema": {
            "type": "object",
            "properties": {
                "holdings": {
                    "type": "object",
                    "description": "Словарь тикер -> количество акций",
                    "additionalProperties": {"type": "number"},
                },
                "prices": {
                    "type": "object",
                    "description": "Словарь тикер -> текущая цена",
                    "additionalProperties": {"type": "number"},
                },
            },
            "required": ["holdings", "prices"],
        },
    },
]


def get_stock_price(ticker: str) -> dict:
    mock_prices = {"AAPL": 189.50, "MSFT": 415.20, "NVDA": 875.40}
    price = mock_prices.get(ticker.upper())
    if price is None:
        raise ValueError(f"Неизвестный тикер: {ticker}")
    return {"ticker": ticker.upper(), "price": price, "currency": "USD"}


def calculate_portfolio_value(holdings: dict, prices: dict) -> dict:
    total = sum(
        shares * prices.get(ticker, 0)
        for ticker, shares in holdings.items()
    )
    return {"total_value": round(total, 2), "currency": "USD"}


def execute_tool(name: str, inputs: dict) -> str:
    try:
        if name == "get_stock_price":
            result = get_stock_price(**inputs)
        elif name == "calculate_portfolio_value":
            result = calculate_portfolio_value(**inputs)
        else:
            result = {"error": f"Неизвестный инструмент: {name}"}
        return json.dumps(result, ensure_ascii=False)
    except Exception as e:
        return json.dumps({"error": str(e)}, ensure_ascii=False)


def run_agent(user_query: str) -> str:
    messages = [{"role": "user", "content": user_query}]

    while True:
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=1024,
            tools=tools,
            messages=messages,
        )

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
            return ""

        if response.stop_reason != "tool_use":
            break

        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result_content = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result_content,
                })

        messages.append({"role": "user", "content": tool_results})

    return "Агентный цикл завершился неожиданно"


answer = run_agent(
    "У меня 10 акций AAPL и 5 акций NVDA. Какова стоимость моего портфеля?"
)
print(answer)
```

## Архитектура безопасности инструментов

```
Запрос пользователя
       │
       ▼
  [[llm|LLM]] генерирует блок tool_use
       │
       ▼
  Валидатор входных данных ──► Отклонить, если схема нарушена
       │
       ▼
  Песочница / Слой разрешений
       │
       ├─► Разрешено: выполнить инструмент
       │         │
       │         ▼
       │   Результат внедряется в контекст
       │
       └─► Запрещено: вернуть сообщение об ошибке модели
```

Стратегии изоляции:
- **Инструменты только для чтения**: безопасно вызывать без подтверждения (поиск, чтение файла).
- **Инструменты записи**: требуют подтверждения человеком или явного разрешения через [[mcp]].
- **Деструктивные инструменты**: всегда требуют явного подтверждения пользователя; никогда не выполнять в плотном цикле.

## Ключевые компромиссы

- **Решает:** доступ к данным в реальном времени, действия во внешних системах, структурированные вычисления, расширение возможностей модели без дообучения.
- **Не решает:** внутренние ограничения рассуждений модели — мусор на входе даёт мусор на выходе.
- **Задержка:** каждый вызов инструмента — это round-trip; параллельные вызовы снижают задержку, но усложняют реализацию.
- **Надёжность:** модели иногда галлюцинируют имена инструментов или выдают невалидный JSON. Валидация по схеме обязательна.
- **Стоимость:** разговоры с инструментами длиннее; каждая итерация цикла добавляет токены.
- **Безопасность:** никогда не передавайте необработанный пользовательский ввод напрямую в инструменты, исполняющие код или записывающие в постоянное хранилище.

## Варианты и расширения

**Принудительный вызов инструмента:** установка `tool_choice = {"type": "tool", "name": "..."}` обязывает модель вызвать конкретный инструмент. Полезно для пайплайнов структурированного извлечения.

**Потоковый вызов инструментов:** API Anthropic со стримингом выдаёт события `input_json_delta` по мере генерации JSON аргументов инструмента, что позволяет обновлять UI в реальном времени.

**Делегирование между агентами:** оркестратор-агент вызывает специализированного суб-агента как инструмент (паттерн agent-as-tool). Суб-агент возвращает результат как структурированный вывод.

**[[mcp]] (Model Context Protocol):** открытый стандарт, обобщающий вызов инструментов — определяет транспортно-независимый протокол для предоставления инструментов, ресурсов и промптов любому совместимому LLM-клиенту.

## Практические применения

- **Интерпретаторы кода:** выполнение Python/JavaScript в изолированной среде с отображением результатов.
- **Генерация запросов к базе данных:** модель пишет SQL, инструмент выполняет его и возвращает строки.
- **Агенты для календаря и почты:** чтение/запись событий, создание и отправка писем.
- **Финансовые copilot-системы:** получение цен, вычисление метрик, резюмирование документов — см. также [[rag]] над финансовыми документами.
- **DevOps-агенты:** анализ логов CI, открытие задач в GitHub, запуск деплоев.

## Tool-use как пространство действий агента

Tool-use — мост от языковой модели, которая генерирует текст, к **агенту**, который может действовать в мире. Как только мы признаём, что «решить, какой инструмент вызвать» — это обученная политика, в дело вступает тот же теоретический аппарат, что и в обучении с подкреплением: LLM — политика, вызовы инструментов — действия, наблюдения — возвраты инструментов, награда — выполнение задачи. CoT-рассуждения между вызовами инструментов оказываются **планированием на этапе инференса** в MDP.

Этот взгляд напрямую связывает tool-use с более широкой RL-литературой:

- **[[deep-rl-execution|Глубокий RL для исполнения]]** в количественной торговле построен на том же цикле действие-наблюдение: модель решает «купить этот срез сейчас / подождать / отменить» исходя из текущего состояния стакана, награда — реализованный P&L. Замена торговой политики на LLM-с-инструментами — буквальная подстановка.
- **[[reinforcement-learning|Обучение с подкреплением]]** как дисциплина даёт формальный каркас (MDP, уравнения Беллмана, policy gradients), который превращает «точность tool-use» в одну оптимизируемую цель. Современные reasoning-модели тренируются ровно этим циклом: награждается трасса, заканчивающаяся правильным ответом.

В количественных финансах tool-use делает практичным [[llm-financial-analysis|LLM-анализ финансов]]: модель может вызвать прайсер QuantLib, забрать ставку ЦБ или запустить Python-регрессию как часть подготовки одного ответа, вместо того чтобы делать вид, что посчитала эти числа в весах.

## Связанные темы

- [[rag]] — поиск может быть реализован как инструмент, позволяя модели решать, когда выполнять поиск.
- [[chain-of-thought]] — ReAct чередует шаги рассуждения (CoT) с действиями через инструменты.
- [[mcp]] — стандартизирует определение инструментов и транспорт между провайдерами и клиентами.
- [[speculative-decoding]] — не связан с выбором инструментов, но влияет на скорость шагов генерации между вызовами инструментов.
- [[reinforcement-learning]] — формальный каркас для «решить, какое действие совершить дальше».
- [[deep-rl-execution]] — тот же action-observation-цикл, применённый к исполнению сделок.
- [[llm-financial-analysis]] — продакшн-tool-use для заземлённого финансового анализа.
