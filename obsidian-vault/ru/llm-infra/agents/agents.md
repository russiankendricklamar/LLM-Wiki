# [[llm|LLM]]-агенты

## Предварительные знания

[[llm]], [[tool-use]], [[chain-of-thought]]

## Обзор

[[llm|LLM]]-агент — это система, которая объединяет языковую модель с циклом, обеспечивающим повторяющиеся наблюдение, рассуждение и действие. Вместо единственного прямого прохода от промпта к ответу, агент работает итеративно: модель производит рассуждения и структурированное действие (вызов инструмента), окружение (или слой приложения) выполняет это действие и наблюдает результат, результат подаётся обратно модели, и цикл повторяется до сигнала модели о завершении или срабатывания условия остановки.

Это замыкает обратную связь, которую чистая LLM не может выполнить. Если языковая модель — это функция $f_\theta: \text{промпт} \to \text{ответ}$, то агент — это политика $\pi_\theta$, которая учится отображать наблюдения и историю на действия, итерируя до завершения задачи. Механизм прост — агенты строятся на трёх примитивах: [[tool-use|вызов инструментов]] для структурированных действий, [[chain-of-thought|цепочка мысли]] для прозрачности рассуждений и память для отслеживания состояния между итерациями.

Паттерн ReAct (Yao et al., 2022) формализовал это как чередование **Рассуждения** и **Действия**. Он эмпирически продемонстрировал, что заземление рассуждений во внешних наблюдениях — в отличие от чистой цепочки мысли — улучшает факическую точность и снижает галлюцинации. Современные LLM, дообученные для агентного поведения (Claude, GPT-4, Gemini 2.0), реализуют ReAct нативно в своём обучении.

## Агентный цикл

Базовый агентный цикл состоит из пяти шагов:

1. **Инициализация:** Конструируется начальный промпт с описанием задачи, доступными инструментами и опциональным системным контекстом.
2. **Вызов модели:** Текущее состояние (история бесед + наблюдения) отправляется в LLM. Модель генерирует как текст рассуждений, так и структурированное действие (вызов инструмента или токен FINISH).
3. **Разбор действия:** Из выхода модели извлекаются имя инструмента и аргументы. Валидируется против схемы. Если невалидный JSON или неизвестный инструмент, возвращается ошибка-наблюдение и цикл повторяется.
4. **Выполнение:** Инструмент выполняется в контролируемой среде (песочница, слой разрешений или внешний API). Результат (успех или ошибка) захватывается.
5. **Внедрение наблюдения:** Результат добавляется к истории бесед как сообщение `tool_result`. Счётчик шага увеличивается. Если модель выдала токен FINISH, из результата извлекается финальный ответ и возвращается. Иначе цикл переходит на шаг 2.

Ограниченный цикл добавляет параметр **max_steps** для предотвращения бесконечных циклов; **timeout** предотвращает зависания на медленных инструментах.

**Псевдокод:**

```
function run_agent(задача, инструменты, max_steps=10):
  messages = [{"role": "user", "content": задача}]
  for step = 1 to max_steps:
    response = llm(messages, tools=инструменты)
    messages.append({"role": "assistant", "content": response})
    
    if response.stop_reason == "FINISH":
      return извлечь_финальный_ответ(response)
    
    if response.stop_reason != "tool_use":
      break  // неожиданное завершение
    
    tool_results = []
    for действие in response.tool_calls:
      result = execute_tool(действие.name, действие.args)
      tool_results.append({"tool_use_id": действие.id, "content": result})
    
    messages.append({"role": "user", "content": tool_results})
  
  return "max_steps превышен"
```

## ReAct: Рассуждение и Действие

ReAct чередует рассуждения (естественный язык мысли) и действия (вызовы инструментов). Это адресует ключевое ограничение чистой цепочки мысли: без заземления в наблюдениях, рассуждения CoT неверны — модель генерирует правдоподобные, но необоснованные утверждения. ReAct принуждает модель рассуждать о том, что она наблюдает, а не о том, что она галлюцинирует.

Пусть $T = (r_1, a_1, o_1, r_2, a_2, o_2, \ldots, r_T, a_T, o_T)$ обозначает траекторию чередующихся шагов рассуждения $r_t$, действий $a_t$ (вызовы инструментов) и наблюдений $o_t$ (результаты инструментов). Политика $\pi_\theta$ генерирует оба:

$$r_t, a_t = \pi_\theta(T_{<t})$$

где $T_{<t}$ — история предыдущих шагов. Каждое $a_t$ — вызов инструмента:

$$a_t = (f_t, \mathbf{x}_t), \quad f_t \in \mathcal{F}, \quad \mathbf{x}_t \models \text{schema}(f_t)$$

и каждое наблюдение детерминировано при данном действии:

$$o_t = \text{execute}(a_t)$$

Критически, во время обучения (для моделей дообученных на ReAct), траектории с корректными промежуточными шагами вознаграждаются выше, чем траектории с корректными финальными ответами, но плохими рассуждениями. Это выравнивает рассуждения модели с последовательностью действий.

**Пример траектории:** "Какое население Франции?"
```
Мысль: Мне нужно поискать текущие данные о населении Франции.
Действие: вызвать search_web(query="население Франции 2025")
Наблюдение: Население Франции примерно 68 миллионов на 2025 год.
Мысль: Поиск вернул прямой ответ. Теперь я могу ответить.
Действие: FINISH(answer="Население Франции примерно 68 миллионов.")
```

Без промежуточной Мысли, чистая модель tool-calling может вызвать search_web, получить наблюдение, но затем галлюцинировать ответ, не поддерживаемый наблюдением.

## Вызов инструментов и композиция скиллов

Возможность агента напрямую ограничена его набором инструментов. Инструменты разбиваются по категориям:

- **Извлечение информации:** `search_web`, `fetch_url`, `query_database`. Обеспечивают доступ к заземлённым знаниям.
- **Вычисления:** `python_interpreter`, `calculator`, `sql_executor`. Позволяют символические манипуляции сверх арифметических ограничений LLM.
- **Действие:** `send_email`, `create_issue`, `update_spreadsheet`. Обеспечивают эффекты во внешних системах.
- **Рефлексия:** `list_available_tools`, `check_task_status`. Обеспечивают мета-рассуждение о возможностях.

Композиция инструментов нетривиальна. Модель должна научиться, какие инструменты вызывать в каком порядке. На практике это решается:

1. **Чёткие описания:** Каждый инструмент имеет краткое описание, помогающее модели узнать релевантность. Плохие описания приводят к галлюцинациям инструментов.
2. **Семантическое встраивание:** Некоторые системы встраивают описания инструментов в векторное пространство и извлекают топ-$k$ наиболее релевантных инструментов, сокращая пространство инструментов со 100+ до 5-10 на запрос.
3. **Прогрессивное уточнение:** Многошаговые агенты уточняют своё понимание в ходе вызовов инструментов. Шаг 1 может выполнить широкий поиск; шаг 2 сузить на основе результата.

## Типы памяти в агентах

### Память в контексте

История бесед (все предыдущие сообщения) — основная память. Она ограничена контекстным окном. Для долгоживущих агентов история растёт неограниченно, если не обрезать. Стратегии:

- **Суммаризация:** Периодически суммаризировать старые сообщения в один блок суммаризации, освобождая токены.
- **Скользящее окно:** Сохранять только последние $k$ сообщений, отбрасывая старые шаги.
- **Сжатие:** Квантизировать или дистиллировать историю в ключевые факты перед внедрением в контекст.

### Внешняя память через [[rag|RAG]]

Для агентов, которым нужно вспомнить прошлые запуски или посмотреть знания домена, [[rag|поиск с дополнением]] существенно важен. Векторная база данных хранит прошлые взаимодействия, решения или факты. Когда агент встречает новый запрос, он извлекает релевантный прошлый контекст через семантический поиск.

$$\text{retrieved} = \text{topk\_similarity}(\text{embed}(q), \text{knowledge\_base})$$

### Эпизодическая память

Некоторые агенты ведут структурированные суммаризации прошлых эпизодов (напр., "во вторник мы попробовали подход A и он не сработал; подход B прошёл успешно"). Они хранятся как структурированные записи, не эмбеддинги, и позволяют явное рассуждение о прошлых ошибках.

## Мульти-агентные системы

Сверх единичных агентов, оркестрированные мульти-агентные системы достигают специализированного рассуждения и улучшенной надёжности:

### Оркестратор + Рабочие

Топ-уровневый оркестратор-агент разлагает задачу на подзадачи, отправляя каждую специализированному рабочему агенту. Оркестратор собирает результаты и синтезирует их.

```
Задача пользователя
    │
    ▼
Оркестратор (CoT: "Эта задача требует исследования, анализа и синтеза")
    │
    ├─► Рабочий 1: research-agent
    │        └─► search_web, fetch documents, summarize
    │
    ├─► Рабочий 2: analysis-agent
    │        └─► run code, compute metrics
    │
    └─► Рабочий 3: synthesis-agent
             └─► draft final report
    │
    ▼
Оркестратор агрегирует + финальный ответ
```

Каждый рабочий сам по себе — полный агент с собственным циклом, инструментами и памятью.

### Рефлексия и Дебаты

Мощный мульти-агентный паттерн — запустить агента несколько раз, генерируя разнообразные предложения, затем имеет агентов критиковать и уточнять каждое предложение:

1. **Предложить:** Агент A генерирует черновик решения.
2. **Критиковать:** Агент B оценивает предложение, выделяя пробелы или ошибки.
3. **Уточнить:** Агент A пересматривает на основе обратной связи.
4. **Консенсус:** Запустить несколько агентов параллельно, агрегировать через голосование или консенсус.

Это отражает человеческую рецензию и значительно улучшает качество решения на сложных задачах рассуждения.

### Параллелизм и стоимость

Мульти-агентная оркестрация торгует увеличенной сложностью за улучшенную надёжность и качество решения. Стоимость — более высокое использование токенов (несколько агентов) и задержка (последовательные циклы уточнения). Для высокорисковых задач (напр., финансовый анализ, медицинское рассуждение), этот компромисс благоприятен.

## Планирование и Tree of Thoughts

Вместо линейного агентного цикла, некоторые агенты изучают несколько путей рассуждения и выбирают наиболее многообещающие. Это паттерн **Tree of Thoughts (ToT)** (Yao et al., 2023).

На каждом шаге $t$ агент генерирует $b$ кандидатов следующих действий (branching factor). Функция значения $V(s_t)$ оценивает многообещание каждой ветви:

$$V(s_t) = \text{LLM-evaluate}(s_t, \text{"Является ли этот шаг многообещающим?"})$$

Алгоритмы поиска (BFS, DFS, beam search) проходят дерево, расширяя высокозначимые узлы. Глубина $d$ и branching factor $b$ определяют полное количество вызовов модели: примерно $O(b^d)$. На сложных головоломках, ToT с $b=3, d=5$ использует намного больше инференса, но находит правильные решения, где линейное рассуждение застревает.

**Компромисс:** ToT дорог (много вызовов модели), но эффективен для NP-полных или комбинаторных задач. Линейные агентные циклы дёшевы, но могут застрять в локальных оптимумах.

## Режимы отказа и вызовы

### Галлюцинированные вызовы инструментов

Модели иногда придумывают имена инструментов или параметры, не в схеме. Смягчение:

- Валидировать все выходы против объявленной схемы перед выполнением.
- Возвращать сообщения об ошибках, на которых модель может учиться: "Инструмент 'search_internet' не существует. Доступные инструменты: search_web, search_scholar, search_arxiv."
- Дообучить модели на датасетах tool-calling для снижения галлюцинаций.

### Бесконечные циклы

Агент может войти в цикл: вызвать инструмент A → получить результат → вызвать инструмент A снова → получить тот же результат. Агент должен научиться выходить. Смягчение:

- **Max steps:** Жёсткий лимит на итерации.
- **Обнаружение цикла:** Отслеживать недавние вызовы инструментов и флагировать повторы.
- **Штраф:** Возвращать ошибки, когда один и тот же инструмент вызывается дважды с теми же аргументами.

### Переполнение контекста

По мере работы агентного цикла, история бесед растёт, потребляя токены. Модель с контекстом 4K переполнится после ~40-50 агентных шагов (предполагая ~100 токенов за раунд). Смягчение:

- Суммаризировать историю периодически.
- Использовать модели с большим контекстом (100K+ для глубокого рассуждения).
- Сжимать наблюдения в факты, а не сырой текст.

### Распространение ошибок

Единственный плохой результат инструмента может каскадировать: модель неправильно интерпретирует результат, вызывает неправильный инструмент, получает другой плохой результат, усложняет ошибку. Смягчение:

- Валидация: Двойная проверка выходов инструментов на согласованность.
- Восстановление: Обеспечить явную обработку ошибок — если инструмент не срабатывает, предложить альтернативы агенту.
- Откат: Позволить агентам переповторять с разными инструментами, когда шаг не срабатывает.

## MCP и Модульные инструменты

[[mcp|Model Context Protocol]] стандартизирует определение инструментов и транспорт. Вместо построения пользовательских схем инструментов для каждого приложения, MCP-серверы предоставляют инструменты через стандартный JSON-RPC протокол. Любой совместимый с MCP агент может подключиться к любому MCP-серверу.

Преимущество: Единственный MCP-сервер, обёрнутый вокруг базы данных компании, может служить агенту на основе Claude, агенту на основе GPT-4 или внутридомовой модели без переиспользования интеграционного кода.

## Текущая граница и надёжность

Продакшн-агенты (Claude Code, GPT-4 Code Interpreter, AutoGPT) работают на практике для многих задач, но **надёжность остаётся нерешённой**. Доминирующий режим отказа — поведение вне распределения: агент хорошо работает на внутри-распределении задач, но катастрофически не срабатывает на edge cases.

Недавние направления исследований:

- **Reasoning модели (o1/o3):** Выделить больше инференс-вычислений на планирование, снижая галлюцинации и улучшая step-wise корректность.
- **Process reward models:** Вознаградить промежуточные шаги (не только финальные ответы), позволяя агентам, обученным через RL, учиться лучшим последовательностям действий.
- **Tool-use fine-tuning:** Контролируемые датасеты экспертных траекторий значительно улучшают надёжность агента.
- **Симуляция и execution sandboxes:** Тестировать поведение агента в симуляторе перед развёртыванием снижает риск развёртывания.

## Пример реализации

Минимальный Python-агент, использующий API Anthropic:

```python
import anthropic
import json

client = anthropic.Anthropic()

tools = [
    {
        "name": "search_web",
        "description": "Поиск информации в веб.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Запрос поиска"}
            },
            "required": ["query"]
        }
    },
    {
        "name": "calculator",
        "description": "Выполнить арифметические операции.",
        "input_schema": {
            "type": "object",
            "properties": {
                "operation": {"type": "string", "description": "add, subtract, multiply, divide"},
                "a": {"type": "number"},
                "b": {"type": "number"}
            },
            "required": ["operation", "a", "b"]
        }
    }
]

def execute_tool(name: str, args: dict) -> str:
    if name == "search_web":
        # Мок-поиск; в продакшне вызовите реальный API
        query = args["query"]
        return f'Результаты поиска для "{query}": [результат 1, результат 2]'
    elif name == "calculator":
        operation = args.get("operation")
        a = args.get("a")
        b = args.get("b")
        results = {
            "add": a + b,
            "subtract": a - b,
            "multiply": a * b,
            "divide": a / b if b != 0 else "Ошибка: деление на ноль"
        }
        result = results.get(operation, "Неизвестная операция")
        return f"Результат: {result}"
    return f"Неизвестный инструмент: {name}"

def run_agent(task: str, max_steps: int = 10) -> str:
    messages = [{"role": "user", "content": task}]
    
    for step in range(max_steps):
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=1024,
            tools=tools,
            messages=messages
        )
        
        messages.append({"role": "assistant", "content": response.content})
        
        # Проверка завершения
        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
            return "Нет ответа"
        
        if response.stop_reason != "tool_use":
            return "Неожиданное завершение"
        
        # Выполнить инструменты
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })
        
        messages.append({"role": "user", "content": tool_results})
    
    return "Max_steps превышен"

# Пример
answer = run_agent("Какой результат 2024 в квадрате? Используй инструмент calculator.")
print(answer)
```

## Практические применения

- **Исследовательские ассистенты:** Рекурсивно искать, синтезировать и проверять факты в статьях.
- **Генерация кода:** Генерировать код, запускать его, отлаживать ошибки, итерировать до прохождения тестов.
- **Финансовый анализ:** Получать рыночные данные, запускать backtests, генерировать отчёты.
- **Автоматизация задач:** Разбирать письма, извлекать action items, создавать события календаря.
- **Чат над базой знаний:** Извлекать документы, синтезировать ответы, предоставлять источники.

## Связанные темы

- [[tool-use]] — примитив действия, который агенты используют для взаимодействия с миром.
- [[chain-of-thought]] — рассуждение внутри агентов; ReAct чередует CoT с действиями.
- [[rag]] — внешнее извлечение знаний, часто интегрированное в агентные циклы через инструмент поиска.
- [[mcp]] — стандартизирует определение инструментов и транспорт для агентов между провайдерами.
- [[reinforcement-learning]] — формальный каркас для обучения политикам агентов (выбор действий).
- [[llm]] — основная модель; большие, лучше-обученные модели составляют лучших агентов.
