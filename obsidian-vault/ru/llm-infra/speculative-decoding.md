# Спекулятивное декодирование

## Обзор

Спекулятивное декодирование — техника ускорения инференса, обеспечивающая 2–4-кратное ускорение авторегрессионной генерации токенов без какого-либо снижения качества. Основная идея проста: быстрая, дешёвая «черновая» (draft) модель спекулятивно генерирует короткую последовательность токенов-кандидатов, а большая «целевая» (target) модель верифицирует их все за один параллельный прямой проход вместо того, чтобы генерировать каждый из них по очереди.

Техника была независимо представлена в 2022–2023 годах: Leviathan et al. (Google Brain) в работе «Fast Inference from Transformers via Speculative Decoding» и Chen et al. (DeepMind) в «Accelerating Large Language Model Decoding with Speculative Sampling». В обеих работах доказывается, что схема принятия/отклонения токенов точно сохраняет распределение токенов целевой модели — то есть деградации качества нет по сравнению со стандартной жадной или температурной выборкой из целевой модели.

Узкое место, которое устраняет спекулятивное декодирование, хорошо изучено: большие авторегрессионные модели при генерации одной последовательности ограничены пропускной способностью памяти, а не вычислительной мощностью. Каждый токен требует загрузки всех весов модели из HBM в вычислительные блоки. Модель с 70B параметрами в fp16 требует загрузки ~140 ГБ на каждый шаг генерации. Объединяя верификацию $k$ черновых токенов в один прямой проход, стоимость передачи памяти амортизируется по $k$ токенам вместо одного.

## Как это работает

**Шаг 1 — Черновая генерация:** маленькая черновая модель $M_q$ авторегрессивно генерирует $k$ токенов: $\tilde{x}_1, \tilde{x}_2, \ldots, \tilde{x}_k$ из текущего контекста. Черновая модель быстрая, потому что маленькая (например, 7B против 70B у целевой).

**Шаг 2 — Параллельная верификация:** целевая модель $M_p$ делает один прямой проход по полному контексту плюс все $k$ черновых токенов одновременно, вычисляя распределения вероятностей $p(\cdot \mid \text{контекст}, \tilde{x}_{<i})$ для каждой позиции $i$ параллельно.

**Шаг 3 — Принятие/отклонение:** для каждого чернового токена $\tilde{x}_i$ сравниваем вероятность целевой модели $p(\tilde{x}_i)$ с вероятностью черновой $q(\tilde{x}_i)$. Принимаем с вероятностью $\min\!\left(1, \frac{p(\tilde{x}_i)}{q(\tilde{x}_i)}\right)$. При отклонении сэмплируем скорректированный токен из поправочного распределения и останавливаемся.

**Шаг 4 — Бонусный токен:** после принятия всех $k$ черновых токенов вывод целевой модели в позиции $k+1$ используется как дополнительный бесплатный токен (целевая уже вычислила его в ходе верификации).

**Эффективная пропускная способность:** если в среднем из $k$ черновых токенов принимается $\alpha$ (коэффициент принятия), каждый «раунд» производит $\alpha k + 1$ токен ценой одного прямого прохода целевой плюс $\alpha k$ прямых проходов черновой. При дешёвой черновой генерации это существенно быстрее, чем $\alpha k + 1$ отдельных проходов целевой модели.

## Математическое описание

Пусть $p(x \mid \text{ctx})$ — целевое распределение, $q(x \mid \text{ctx})$ — черновое распределение. Схема принятия/отклонения сэмплирует из модифицированного распределения:

$$
\tilde{p}(x) = \begin{cases}
q(x) & \text{с вероятностью } \min\!\left(1, \frac{p(x)}{q(x)}\right) \\
\text{отклонить, пересэмплировать из } p'(x) & \text{иначе}
\end{cases}
$$

где поправочное остаточное распределение:

$$p'(x) = \frac{\max(0,\; p(x) - q(x))}{\sum_{x'} \max(0,\; p(x') - q(x'))}$$

Ключевая теорема (Leviathan et al.) доказывает, что результирующее распределение принятых токенов точно равно $p(x)$:

$$\mathbb{E}[\tilde{p}(x)] = p(x) \quad \forall x$$

Это означает, что спекулятивное декодирование является точной аппроксимацией — не сжатием и не дистилляцией. Оно ускоряет сэмплирование, не изменяя того, что сэмплируется.

Ожидаемое число принятых черновых токенов за раунд:

$$\mathbb{E}[\alpha] = \sum_{x} \min(p(x), q(x)) = 1 - \text{TV}(p, q)$$

где $\text{TV}(p, q)$ — расстояние полного вариирования между целевым и черновым распределениями. Чем ближе черновое к целевому, тем выше коэффициент принятия и тем больше ускорение.

Приблизительный коэффициент ускорения $S$:

$$S \approx \frac{1 + \mathbb{E}[\alpha]}{1 + c \cdot \mathbb{E}[\alpha]}$$

где $c = t_{\text{draft}} / t_{\text{target}}$ — отношение времени генерации на токен черновой к целевой. При $c \approx 0.1$ и $\mathbb{E}[\alpha] = 3$, $S \approx 3.6\times$.

## Реализация

Следующий псевдокод отражает цикл принятия/отклонения в ядре спекулятивного декодирования. На практике это реализуется внутри фреймворка для обслуживания моделей (vLLM, TGI, TensorRT-LLM), а не предоставляется напрямую в прикладной код.

```python
import torch
import torch.nn.functional as F


def speculative_decode_step(
    context_ids: torch.Tensor,
    target_model,
    draft_model,
    k: int = 4,
    temperature: float = 1.0,
) -> torch.Tensor:
    """
    Один раунд спекулятивного декодирования.
    Возвращает новые токены для добавления к context_ids.

    Аргументы:
        context_ids:  (1, seq_len) тензор токенов текущего контекста.
        target_model: большая верифицирующая модель (например, 70B).
        draft_model:  маленькая черновая модель (например, 7B).
        k:            количество черновых токенов за раунд.
        temperature:  температура сэмплирования.

    Возвращает:
        accepted_tokens: (1, m) тензор, где 1 <= m <= k+1.
    """
    draft_ids = context_ids.clone()
    draft_token_list: list[torch.Tensor] = []
    draft_prob_list: list[float] = []

    # Фаза 1: черновая модель авторегрессивно генерирует k токенов
    with torch.no_grad():
        for _ in range(k):
            out = draft_model(draft_ids)
            logits = out.logits[:, -1, :] / max(temperature, 1e-5)
            probs = F.softmax(logits, dim=-1)
            token = torch.multinomial(probs, num_samples=1)
            draft_token_list.append(token)
            draft_prob_list.append(probs[0, token.item()].item())
            draft_ids = torch.cat([draft_ids, token], dim=-1)

    # Фаза 2: целевая модель верифицирует все k токенов ОДНИМ прямым проходом
    verify_ids = torch.cat([context_ids] + draft_token_list, dim=-1)
    with torch.no_grad():
        target_out = target_model(verify_ids)

    ctx_len = context_ids.shape[1]
    target_logits = target_out.logits[0, ctx_len - 1:, :]
    target_probs_all = F.softmax(target_logits / max(temperature, 1e-5), dim=-1)

    # Фаза 3: принятие/отклонение каждого чернового токена
    accepted: list[torch.Tensor] = []
    for i, (tok, q_prob) in enumerate(zip(draft_token_list, draft_prob_list)):
        p_prob = target_probs_all[i, tok.item()].item()
        accept_prob = min(1.0, p_prob / (q_prob + 1e-8))
        if torch.rand(1).item() < accept_prob:
            accepted.append(tok)
        else:
            # Пересэмплируем из остаточного распределения
            draft_logits = draft_model(context_ids).logits[0, -1, :]
            draft_probs = F.softmax(draft_logits / max(temperature, 1e-5), dim=-1)
            residual = torch.clamp(target_probs_all[i] - draft_probs, min=0.0)
            residual = residual / residual.sum().clamp(min=1e-8)
            corrected = torch.multinomial(residual.unsqueeze(0), num_samples=1)
            accepted.append(corrected)
            break  # останавливаемся при первом отклонении

    # Фаза 4: бонусный токен — целевая уже вычислила его бесплатно
    bonus_pos = len(accepted)
    bonus = torch.multinomial(
        target_probs_all[bonus_pos].unsqueeze(0), num_samples=1
    )
    accepted.append(bonus)

    return torch.cat(accepted, dim=-1)
```

Производственная заметка: фреймворки **vLLM** (v0.4+) и **TensorRT-LLM** включают оптимизированные ядра спекулятивного декодирования. Приведённый выше код отражает алгоритм; для реальных развёртываний используйте эти библиотеки.

## Варианты черновых моделей

| Стратегия черновика | Описание | Коэффициент принятия | Накладные расходы |
|---|---|---|---|
| Меньшая модель того же семейства | Llama-7B черновик для Llama-70B | 0,70–0,85 | Низкие |
| N-граммная модель | Вероятные продолжения из кэша промпта | 0,50–0,70 | Минимальные |
| Головы MEDUSA | Дополнительные FFN-головы поверх целевой | 0,65–0,80 | Внутрипроцессные |
| EAGLE | Черновик на основе скрытых состояний целевой | 0,80–0,90 | Умеренные |

**MEDUSA** (Cai et al., 2024) присоединяет несколько «голов» к замороженной целевой модели, каждая из которых предсказывает токены на $k$ шагов вперёд. Это устраняет необходимость в отдельной черновой модели, но требует дообучения дополнительных голов.

**EAGLE** (Li et al., 2024) использует предпоследние скрытые состояния целевой модели как входные данные для черновой, достигая очень высокого совпадения распределений.

## Ключевые компромиссы

- **Нулевая потеря качества:** математически гарантированное совпадение с целевым распределением — не аппроксимация.
- **Зависимость от оборудования:** ускорение наиболее выражено на оборудовании, ограниченном пропускной способностью памяти (A100, H100). TPU с другой иерархией памяти могут показывать меньший выигрыш.
- **Чувствительность к размеру батча:** спекулятивное декодирование эффективно для одной последовательности или малых батчей. Большой батч-инференс уже ограничен вычислениями, поэтому выигрыш уменьшается.
- **Обслуживание черновой модели:** использование отдельной черновой модели добавляет операционную сложность (две версии модели для совместного развёртывания и обновления).
- **Чувствительность к промпту:** коэффициент принятия варьируется в зависимости от типа текста. Код и структурированный текст, как правило, имеют более высокий коэффициент; творческий текст с широким разнообразием токенов — более низкий.

## Варианты и расширения

**Само-спекулятивное декодирование:** целевая модель сама генерирует черновики, пропуская определённые слои (layer-skip drafting), устраняя необходимость в отдельной черновой модели.

**Пакетное спекулятивное декодирование:** расширяет алгоритм на батч-инференс, поддерживая общую структуру верификации по элементам батча.

**SpecInfer:** расширяет спекулятивное декодирование до древовидных черновиков, одновременно предлагая несколько продолжений и верифицируя их все за один tree-attention прямой проход.

**DistillSpec:** использует дистилляцию знаний для обучения черновой модели специально имитировать распределение следующего токена целевой, максимизируя коэффициент принятия.

## Практические применения

- **Генерация длинных текстов:** исследовательские резюме, генерация кода — чем длиннее выход, тем больше накапливается ускорение. Особенно ценно для chain-of-thought рассуждений, где генерируются сотни токенов.
- **Интерактивные ассистенты:** спекулятивное декодирование снижает задержку первого токена и задержку на токен, улучшая воспринимаемую отзывчивость.
- **Инференс на устройстве:** мобильные и периферийные развёртывания, где крошечная черновая модель на устройстве предлагает токены, которые облачная целевая модель верифицирует за один вызов, амортизируя стоимость сетевого round-trip.
- **Снижение стоимости:** облачное ценообразование инференса часто исчисляется за вычислительные токены. Спекулятивное декодирование может снизить выставляемые вычисления для длинных выходов.

## Место в более широком стеке инференса

Спекулятивное декодирование — один из трёх ортогональных рычагов более дешёвого / быстрого инференса трансформеров, каждый из которых эксплуатирует разные узкие места базовой [[transformer-architecture|трансформер-архитектуры]]:

- **Архитектурный уровень**: [[mixture-of-experts|Mixture-of-Experts]] активирует лишь часть весов модели на каждый токен, обменивая количество параметров на вычисления на токен. Спекулятивное декодирование полностью совместимо с MoE — оба рычага можно складывать.
- **Алгоритмический уровень**: спекулятивное декодирование амортизирует стоимость memory-bandwidth по $k$ кандидатам при том же объёме переноса весов.
- **Уровень compute-экономики**: [[neural-scaling-laws|нейронные scaling laws]] диктуют оптимальное отношение compute / параметров при заданном бюджете. Спекулятивное декодирование сдвигает кривую стоимости на конце инференса, не требуя переобучения.

Эти три рычага обычно комбинируются в продакшн-стеках: MoE-целевая модель верифицирует черновики, выданные плотной малой моделью, на оборудовании, где пропускная способность памяти — то самое связывающее ограничение, ради обхода которого и придумано спекулятивное декодирование.

Для [[chain-of-thought|моделей с расширенным мышлением]], выпускающих сотни внутренних reasoning-токенов до любого пользовательского вывода, спекулятивное декодирование вдвойне ценно — ускорение масштабируется линейно с длиной рассуждения, поэтому продакшн-reasoning-модели (Claude extended thinking, o1/o3) сильно на него опираются.
