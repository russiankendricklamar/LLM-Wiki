# MCP (Model Context Protocol)

## Обзор

Model Context Protocol (MCP) — открытый стандарт, представленный Anthropic в ноябре 2024 года, определяющий унифицированный интерфейс для подключения языковых моделей к внешним инструментам, источникам данных и сервисам. Там где специфический [[tool-use]] требует, чтобы каждое приложение определяло собственные схемы функций и слой выполнения, MCP устанавливает нейтральный к провайдеру протокол, позволяющий любому совместимому LLM-клиенту подключаться к любому совместимому MCP-серверу без специфической интеграции.

Аналогия, используемая Anthropic, весьма показательна: MCP относится к вызову инструментов LLM так же, как LSP (Language Server Protocol) относится к редакторам кода. До LSP каждый редактор строил собственную интеграцию с каждым языковым тулчейном. После LSP единый протокол позволил любому редактору общаться с любым языковым сервером. MCP стремится к той же компонуемости в экосистеме языковых моделей.

Практическая проблема, которую решает MCP, — фрагментация. По мере роста числа LLM-приложений разработчики строят изолированные инструментальные интеграции: Claude получает инструмент чтения файлов, GPT-4 — свой, Gemini — ещё один. Каждая интеграция дублирует управление доступом, определение схем, транспортный код и обработку ошибок. MCP выносит всё это в сервер, к которому может подключиться любой клиент. Единый MCP-сервер, предоставляющий доступ к корпоративной базе данных, может быть подключён к любому агенту, плагину для IDE или чат-интерфейсу, поддерживающему MCP.

## Архитектура

MCP определяет три роли: **хост** (host), **клиент** (client) и **сервер** (server).

- **Хост** — приложение (например, Claude.ai, расширение VS Code, кастомный агент). Управляет одним или несколькими клиентскими подключениями.
- **Клиент** — экземпляр MCP-клиента, встроенный в хост. Поддерживает соединение 1:1 с MCP-сервером и управляет согласованием протокола.
- **Сервер** — процесс (или удалённый сервис), предоставляющий возможности: инструменты, ресурсы и промпты.

```
┌─────────────────────────────────────────────┐
│              ХОСТ-ПРИЛОЖЕНИЕ                │
│                                             │
│  ┌──────────┐    ┌────────────────────┐    │
│  │  LLM API │    │   MCP Клиент 1     │────┼──► MCP Сервер A (файловая система)
│  │ (Claude) │    │   MCP Клиент 2     │────┼──► MCP Сервер B (база данных)
│  └──────────┘    │   MCP Клиент 3     │────┼──► MCP Сервер C (веб-поиск)
│                  └────────────────────┘    │
└─────────────────────────────────────────────┘
```

Такая архитектура означает, что хост может объединять несколько MCP-серверов одновременно. LLM видит единое пространство инструментов, агрегированное со всех подключённых серверов.

## Примитивы

MCP предоставляет четыре типа примитивов:

**Инструменты (Tools)** — вызываемые функции. Они следуют определению JSON Schema (идентичному по структуре схемам function calling OpenAI/Anthropic) и возвращают структурированные результаты.

**Ресурсы (Resources)** — адресуемые блобы данных, идентифицируемые URI (например, `file:///path/to/doc.txt`, `db://table/row_id`). Ресурсы получаются клиентом по запросу и внедряются в контекст. Это разделяет доступ на чтение (ресурсы) и действия (инструменты).

**Промпты (Prompts)** — повторно используемые, параметризованные шаблоны промптов, публикуемые серверами. Клиент может вызвать именованный промпт с аргументами и получить подготовленный список сообщений для прямого внедрения в беседу с LLM.

**Сэмплирование (Sampling)** — опциональная возможность, при которой MCP-сервер просит клиент запустить LLM-инференс от его имени. Это обеспечивает рекурсивные агентные паттерны, где инструменты сами нуждаются в доступе к языковой модели.

## Транспортный уровень

MCP поддерживает два транспорта:

**stdio:** хост запускает MCP-сервер как дочерний процесс и общается через stdin/stdout посредством JSON-RPC 2.0 сообщений, разделённых символом новой строки. Идеально для локальных инструментов, разработки и безопасной изоляции.

**HTTP + SSE (Server-Sent Events):** сервер слушает HTTP-эндпоинт. Клиент отправляет запросы через HTTP POST и получает стримируемые ответы через SSE. Подходит для удалённых серверов, облачных сервисов и сценариев с несколькими клиентами.

Оба транспорта используют одинаковый формат сообщений JSON-RPC 2.0, поэтому смена транспорта — изменение конфигурации, а не кода.

## Математическое описание

Согласование возможностей MCP следует модели пересечения множеств возможностей. Пусть $C_{\text{client}}$ — множество возможностей, объявленных клиентом, $C_{\text{server}}$ — возможности, предлагаемые сервером. Активные возможности:

$$C_{\text{active}} = C_{\text{client}} \cap C_{\text{server}}$$

Выбор инструмента LLM из пространства инструментов $\mathcal{T}$ всех подключённых серверов:

$$\mathcal{T} = \bigcup_{s \in \mathcal{S}} \mathcal{T}_s$$

где $\mathcal{S}$ — множество активных MCP-серверов, вероятность выбора инструмента моделью:

$$p(t \mid q) \propto \exp\!\left(\text{relevance}(t, q)\right), \quad t \in \mathcal{T}$$

Модель безопасности использует **управление доступом на основе возможностей**: сервер предоставляет клиенту только те возможности, которые явно объявил при рукопожатии. Расширение возможностей после согласования невозможно без нового подключения.

## Реализация

```python
"""
Минимальный MCP-сервер с использованием официального mcp Python SDK.
Предоставляет два инструмента: read_file и list_directory.
Запуск: python mcp_server.py
Подключение: любой MCP-клиент (например, Claude Desktop, библиотека mcp-client).
"""

import os
import json
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent, CallToolResult

app = Server("filesystem-server")


@app.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="read_file",
            description="Прочитать содержимое файла по указанному пути.",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "Абсолютный или относительный путь к файлу.",
                    }
                },
                "required": ["path"],
            },
        ),
        Tool(
            name="list_directory",
            description="Перечислить все файлы в директории.",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "Путь к директории.",
                    }
                },
                "required": ["path"],
            },
        ),
    ]


@app.call_tool()
async def call_tool(name: str, arguments: dict) -> CallToolResult:
    if name == "read_file":
        file_path = arguments["path"]
        if not os.path.isfile(file_path):
            return CallToolResult(
                content=[TextContent(type="text", text=f"Файл не найден: {file_path}")]
            )
        with open(file_path, encoding="utf-8") as f:
            content = f.read()
        return CallToolResult(content=[TextContent(type="text", text=content)])

    if name == "list_directory":
        dir_path = arguments["path"]
        if not os.path.isdir(dir_path):
            return CallToolResult(
                content=[TextContent(type="text", text=f"Не директория: {dir_path}")]
            )
        entries = os.listdir(dir_path)
        return CallToolResult(
            content=[TextContent(type="text", text=json.dumps(entries, ensure_ascii=False))]
        )

    return CallToolResult(
        content=[TextContent(type="text", text=f"Неизвестный инструмент: {name}")]
    )


if __name__ == "__main__":
    import asyncio
    asyncio.run(stdio_server(app))
```

Для подключения этого сервера к Claude Desktop добавьте в `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "python",
      "args": ["/path/to/mcp_server.py"]
    }
  }
}
```

## MCP против прямого Function Calling

| Аспект | Прямой Function Calling | MCP |
|---|---|---|
| Стандартизация | Схема на каждого провайдера | Унифицированный JSON-RPC протокол |
| Переиспользование | Для каждого приложения | Общий для любого клиента |
| Транспорт | Внутрипроцессный / HTTP | stdio или HTTP+SSE |
| Мультипровайдерность | Нужна переимплементация | Один сервер, любой клиент |
| Доступ к ресурсам | Нестандартный | Первоклассный примитив `resources` |
| Компонуемость | Ограниченная | Несколько серверов объединяются |

## Ключевые компромиссы

- **Решает:** фрагментацию инструментальных интеграций, дублирование управления доступом, привязку к конкретному провайдеру в определениях инструментов.
- **Не решает:** фундаментальную проблему надёжности — языковые модели по-прежнему галлюцинируют имена инструментов или выдают некорректные входные данные.
- **Накладные расходы:** MCP добавляет subprocess или сетевой хоп по сравнению с внутрипроцессными вызовами. Для stdio MCP это ~1 мс; для удалённого MCP — время сетевого round-trip.
- **Безопасность:** MCP-серверы работают с правами хост-процесса. Скомпрометированный MCP-сервер может действовать от имени хоста. Изоляция MCP-серверов — вопрос конфигурации развёртывания.
- **Версионирование:** версионирование протокола MCP обрабатывается при согласовании рукопожатия. Серверы должны корректно обрабатывать согласование возможностей при работе со старыми клиентами.

## Варианты и расширения

**Удалённый MCP через HTTP:** Cloudflare, AWS и другие предлагают хостинг MCP-серверов, позволяя создавать бессерверные инструментальные эндпоинты, доступные с любого MCP-клиента.

**MCP с OAuth:** спецификация включает поток авторизации для делегированного доступа пользователей, позволяя MCP-серверам действовать от имени аутентифицированных пользователей (например, MCP для Google Drive с OAuth-токенами пользователя).

**Мультиагентный MCP:** примитивы сэмплирования позволяют MCP-серверам вызывать LLM-рассуждения, создавая рекурсивные агентные архитектуры. Агент-оркестратор может делегировать задачи специализированным суб-агентам, предоставляемым как MCP-серверы.

**Реальные MCP-серверы (экосистема сообщества):**
- `filesystem` — чтение/запись локальных файлов.
- `sqlite` — запросы и модификация SQLite-баз данных.
- `github` — репозитории, задачи, PR.
- `puppeteer` — веб-браузинг и парсинг.
- `slack` — отправка сообщений, чтение каналов.
- `rag-server` — оборачивает [[rag]]-бэкенд поиска как MCP-ресурс/инструмент.

## Практические применения

- **IDE-ассистенты:** VS Code или Cursor подключаются к MCP-серверам для поиска по кодовой базе, запуска тестов и линтинга — одни и те же серверы работают независимо от того, какая LLM управляет ассистентом.
- **Корпоративный доступ к знаниям:** единый MCP-сервер для Confluence/Notion предоставляет документы как ресурсы; любой совместимый чат-интерфейс может их запрашивать.
- **DevOps-агенты:** MCP-серверы для Kubernetes, Datadog, PagerDuty — агент может диагностировать и устранять инциденты, компонуя эти серверы.
- **Рабочие процессы data science:** MCP-серверы для доступа к базам данных, выполнения Python (в изоляции) и построения графиков создают унифицированный интерфейс для аналитических агентов.
