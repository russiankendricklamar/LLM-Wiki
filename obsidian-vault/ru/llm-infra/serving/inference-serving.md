Инференс больших языковых моделей в продакшене — это задача доставки предсказаний с минимальной латентностью и максимальной пропускной способностью при ограниченных ресурсах памяти и вычислений. Ключевая задача: KV-кэш памяти растёт с каждым сгенерированным токеном, быстро становясь узким местом. Для модели с 7B параметров на контексте 4096 токенов KV-кэш занимает примерно 1 ГБ памяти — инференс становится **memory-bound**, а не compute-bound. Классические методы батчинга неэффективны: когда разные запросы завершаются в разное время, GPU простаивает. Современные системы инференса (vLLM, TensorRT-[[llm|LLM]], TGI) решают эту задачу через **PagedAttention** и **continuous batching**, достигая 3-4× увеличения пропускной способности.

## KV-кэш: компромисс памяти и вычислений

Во время автогрессивной генерации модель обрабатывает один новый токен за шаг. Без кэширования это требует $O(L^2)$ операций на шаг, где $L$ — длина последовательности: нужно пересчитать внимание для всех прошлых токенов вновь. С KV-кэшем сохраняются матрицы $K$ и $V$ для всех предыдущих токенов; каждый новый токен требует только $O(L)$ операций (один forward pass), но память растёт линейно.

Память на KV-кэш: $M = 2 \times n_{\text{layers}} \times n_{\text{tokens}} \times d_{\text{head}} \times \text{batch\_size} \times 4\text{ bytes}$

Для модели 7B ($n_{\text{layers}} = 32$, $d_{\text{head}} = 128$) с одним запросом на 4096 токенов это ~256 МБ; с батчем из 16 запросов — 4 ГБ. На A100 с 80 ГБ VRAM вмещается максимум ~15-20 одновременных запросов без фрагментации. Классический батчинг требует, чтобы все запросы в батче заканчивались одновременно; если один запрос коротче других, выделенная память для его KV-кэша простаивает.

## PagedAttention: виртуальная память для кэша

PagedAttention (Kwon et al., 2023, vLLM) переносит идею виртуальной памяти из операционных систем на KV-кэш. Вместо хранения KV-кэша как сплошного блока, он разбивается на страницы фиксированного размера (обычно 128 токенов). **Таблица блоков** отображает логические позиции на физические адреса в памяти; блоки не обязаны быть смежными.

**Ключевые преимущества:**

1. **Элиминация фрагментации**: когда запрос завершается, его страницы помечаются как свободные и переиспользуются; нет накопления "мёртвого" кэша между запросами.
2. **Общее использование кэша**: запросы с общим префиксом (например, few-shot prompt) могут делиться страницами; при перенаправлении лучей (beam search) дочерние последовательности делят префиксный кэш родителя, требуя только копии блок-таблицы (O(1) вместо O(L)).
3. **3-4× рост пропускной способности**: с PagedAttention вмещается 50+ одновременных запросов против 15-20 без оптимизации.

Фьюжд кернел attention тривиально обобщается на страничную память: блок-таблица передаётся в кернел, который индексирует страницы динамически.

## Continuous batching: асинхронный конвейер

Статическое батчирование требует синхронизации: новый батч формируется только когда все запросы текущего батча завершены. Это приводит к недостатку GPU в сценариях с переменной длиной: 16 коротких запросов + 1 длинный на 1000 токенов означают, что GPU ждёт завершения одного запроса.

**Continuous batching** (Orca, 2022) вставляет новые запросы в батч в естественные границы последовательности. Каждый запрос отслеживается независимо в таблице состояния. Когда запрос генерирует токен `[EOS]` или достигает max_tokens, он удаляется; на его место вставляется новый запрос. GPU работает непрерывно на одной и той же батч-размере, достигая **98-99% утилизации**.

Реализация требует:
- **Динамического переупорядочивания индексов**: логические ID запросов отображаются на физические индексы в батче; переупорядочивание в конце каждого шага.
- **Тоннели** (tokens_per_request): отслеживание генерированного количества токенов для SLO enforcement (макс. время в системе).
- **Слияния префиксов**: если новый запрос разделяет префикс с существующим, PagedAttention позволяет избежать дублирования вычислений.

## Фаза prefill vs. decode: асимметричный граф вычисления

Инференс разбивается на два этапа с противоположной характеристикой:

**Prefill** (обработка prompt): $n_{\text{prompt}}$ токенов за один forward pass. Compute-bound; можно использовать большой batch. Однако огромный batch вводит дополнительную латентность ко **time-to-first-token (TTFT)**.

**Decode** (генерация): один токен на forward pass, autoregressive. Memory-bound; пропускная способность ограничена пропускной способностью памяти ([[flash-attention|HBM]] bandwidth). Большой batch здесь почти не добавляет TTFT (все запросы ждут decode anyway), но может превысить память.

**Disaggregация**: некоторые системы разделяют prefill и decode на разные GPU или даже чипы. Prefill GPU может обработать много prompt-heavy запросов параллельно; decode GPU с меньшей ёмкостью памяти обслуживает генерацию. Требует сетевого трафика, но балансирует нагрузку.

## Flash [[attention-mechanisms|Attention]] и длинные контексты

Flash Attention (Dao et al., 2022) переупорядочивает вычисление внимания $\text{softmax}(QK^T/\sqrt{d_k})V$ так, чтобы минимизировать чтения/записи в высокополосную память (HBM). Классический attention требует $O(L^2)$ HBM I/O; Flash Attention достигает $O(L)$ за счёт фьюжд кернела, выполняющего computation на-лету в быстрой памяти ([[flash-attention|SRAM]]).

Для контекстов >4K токенов Flash Attention критичен. На длинных контекстах (32K+) это дает 2-3× ускорение против стандартной реализации.

## Спекулятивное декодирование

**Speculative decoding** параллелизует декодирование: небольшая черновая модель генерирует $k$ токенов, основная модель верифицирует их в одном forward pass. Если сгласие $\alpha$ высокое (например, 95%), ускорение ~$\alpha \times k$. Если распределения draft и target близки (например, quantized версия того же checkpoint), спекулятивное декодирование даёт 1.5-2× ускорение без доп. памяти.

## Ключевые метрики и SLO

- **Tokens/sec/GPU**: глобальная пропускная способность (включает prefill + decode).
- **TTFT (time-to-first-token)**: латентность от submit до первого токена. SLO: <100ms для interactive, <500ms для batch.
- **TBT (time-between-tokens)**: латентность между последующими токенами. SLO: <50ms. Определяет воспринимаемую "fluency".
- **GPU utilization**: % времени, GPU обрабатывает полезный работу. Цель: >95%.
- **Batch size (dynamic)**: варьируется от 1 (один длинный запрос) до 50+ (много коротких) с continuous batching.

Оптимизация требует компромисса: агрессивное батчирование увеличивает TBT, SLO-constrained batching снижает пропускную способность.

## Системы инференса

**vLLM** (Berkeley LMSYS): наиболее популярный open-source, PagedAttention by default, поддержка [[speculative-decoding|спекулятивного декодирования]], distributed inference. Стандарт de facto для research и production.

**TensorRT-LLM** (NVIDIA): proprietary, fused kernels для NVIDIA GPU, лучшая производительность на A100/H100, сложнее настраивать.

**Text Generation Inference** (HuggingFace): сосредоточен на single-GPU latency, хороший для edge, меньше batch-оптимизаций.

**Ollama**: локальное однопроцессное инференс, простота, нет батчирования.

**SGLang**: structured generation, constrained decoding, хорош для JSON/regex output.
