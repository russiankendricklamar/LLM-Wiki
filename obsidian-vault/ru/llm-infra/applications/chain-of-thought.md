# Цепочка мысли (Chain-of-Thought)

## Необходимые знания

[[llm]]

## Обзор

Цепочка мысли (Chain-of-Thought, CoT) — техника промптинга, побуждающая языковые модели формулировать пошаговые промежуточные рассуждения перед тем, как выдать финальный ответ. Вместо перехода напрямую от вопроса к ответу модель направляется — или обучается — чётко прописывать цепочку логических шагов, подобно тому как человек работает с черновиком. Это резко улучшает производительность на задачах, требующих арифметики, символьных преобразований, здравого смысла и многошагового вывода.

Техника была формализована Wei et al. (Google Brain, 2022) в работе «Chain-of-Thought Prompting Elicits Reasoning in Large Language Models». Авторы показали, что включение в few-shot промпт проработанных примеров с явными шагами рассуждений заставляло достаточно большие модели (≥100B параметров на тот момент) обобщать паттерн и решать задачи, с которыми они прежде не справлялись. Это был ключевой результат: возможности могут быть разблокированы промптингом, а не изменениями архитектуры.

В современных инференс-стеках CoT — уже не просто few-shot трюк. Он встроен в обучение моделей через [[rlhf|RLHF]] с трассами рассуждений, в системные промпты для развёрнутых ассистентов и в гибридные фреймворки вроде [[tool-use]] (ReAct), чередующие рассуждения с внешними действиями. Расширенное мышление (extended thinking), реализованное в моделях вроде Claude 3.7 Sonnet, делает цепочку рассуждений самостоятельным артефактом с собственным бюджетом токенов.

## Как это работает

**Стандартный few-shot CoT:** промпт включает $k$ примеров, каждый из которых показывает вопрос, за которым следует цепочка рассуждений и финальный ответ. Модель генерирует аналогичную цепочку для нового вопроса перед тем, как сформулировать ответ.

**Zero-shot CoT:** добавление фразы «Давайте думать пошагово» к промпту — подход, введённый Kojima et al. (2022) — активирует поведение CoT без примеров. Модель воспринимает фразу как инструкцию расширить процесс рассуждений.

**Самосогласованность (Self-consistency):** вместо одного жадного декодирования делается выборка $m$ цепочек рассуждений со стохастичностью, и по финальному ответу берётся голосование большинством. Предложено Wang et al. (2022): рассуждение трактуется как стохастический процесс, что повышает надёжность на задачах с единственным правильным ответом.

**Дерево мыслей (Tree of Thoughts, ToT):** Yao et al. (2023) расширяют CoT от линейной цепочки до древовидной структуры. Модель генерирует несколько вариантов шагов рассуждения в каждом узле, оценивает их перспективность и исследует наиболее обещающие ветви. Это позволяет заглядывать вперёд и откатываться — необходимо для комбинаторных задач.

**Программа мысли (Program of Thought, PoT):** Chen et al. (2022) предлагают модели генерировать исполняемый код как трассу рассуждений вместо естественного языка. Код запускается интерпретатором Python, а результат — финальный ответ. PoT обходит арифметические ошибки и проблемы обобщения длины, присущие рассуждениям на естественном языке.

## Математическое описание

Пусть $\mathcal{X}$ — пространство задач, $\mathcal{Y}$ — пространство ответов. Без CoT модель оценивает $p_\theta(y \mid x)$ напрямую. С CoT цепочки рассуждений $z \in \mathcal{Z}$ вводятся как скрытые переменные:

$$p_\theta(y \mid x) = \sum_{z \in \mathcal{Z}} p_\theta(y \mid x, z) \cdot p_\theta(z \mid x)$$

На практике $z$ аппроксимируется одной жадной цепочкой (стандартный CoT) или $m$ сэмплированными цепочками (самосогласованность). Оценка самосогласованности:

$$\hat{y} = \arg\max_{y} \sum_{i=1}^{m} \mathbf{1}[\text{answer}(z_i) = y], \quad z_i \sim p_\theta(\cdot \mid x)$$

Дерево мыслей формулирует задачу как поиск по графу рассуждений $G = (V, E)$, где узлы — частичные состояния рассуждения, а рёбра — шаги рассуждения. Модель выступает одновременно генератором и оценщиком:

$$V_{\text{expand}}(v) = \text{[[llm|LLM]]-generate}(v), \quad V_{\text{value}}(v) = \text{LLM-evaluate}(v)$$

Алгоритмы поиска (BFS, DFS, beam search) обходят $G$ в поиске пути $v_0 \to v_1 \to \ldots \to v_T$, ведущего к правильному ответу. Глубина $T$ и ветвление $b$ определяют стоимость: $O(b^T)$ вызовов модели.

## Реализация

```python
import anthropic
import re
from collections import Counter

client = anthropic.Anthropic()


def zero_shot_cot(problem: str) -> str:
    """Zero-shot CoT: добавляем 'Давайте думать пошагово' к любой задаче."""
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": f"{problem}\n\nДавайте думать пошагово.",
            }
        ],
    )
    return response.content[0].text


def self_consistent_cot(
    problem: str,
    num_samples: int = 5,
    temperature: float = 0.7,
) -> dict:
    """
    Самосогласованность: сэмплируем N независимых цепочек рассуждений,
    извлекаем финальный ответ из каждой, берём голосование большинством.
    """
    answers = []

    for _ in range(num_samples):
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=1024,
            temperature=temperature,
            messages=[
                {
                    "role": "user",
                    "content": (
                        f"{problem}\n\n"
                        "Рассуди пошагово, затем заверши ответ: "
                        "'Таким образом, итоговый ответ: <ответ>'"
                    ),
                }
            ],
        )
        text = response.content[0].text
        match = re.search(
            r"итоговый ответ[:\s]+(.+?)(?:\n|$)", text, re.IGNORECASE
        )
        if match:
            answers.append(match.group(1).strip())

    if answers:
        vote_counts = Counter(answers)
        majority_answer, votes = vote_counts.most_common(1)[0]
    else:
        majority_answer, votes = "Определить не удалось", 0

    return {
        "majority_answer": majority_answer,
        "vote_count": votes,
        "total_samples": num_samples,
    }


def program_of_thought(problem: str) -> str:
    """
    Программа мысли: модель пишет код как шаг рассуждения.
    Код затем передаётся инструменту (см. паттерн tool-use) для выполнения.
    Возвращает строку сгенерированного кода; выполнение — на стороне вызывающего.
    """
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": (
                    "Напиши Python-выражение или короткий скрипт, "
                    "вычисляющий числовой ответ. Выводи только код.\n\n"
                    f"Задача: {problem}"
                ),
            }
        ],
    )
    code = response.content[0].text
    return re.sub(r"```python\n?|```\n?", "", code).strip()


# Пример
problem = (
    "Поезд едет из города А в город Б со скоростью 60 км/ч. "
    "На обратном пути скорость составляет 40 км/ч. "
    "Какова средняя скорость за всю поездку туда и обратно?"
)

print("Zero-shot CoT:")
print(zero_shot_cot(problem)[:400])

print("\nСамосогласованность (5 сэмплов):")
sc = self_consistent_cot(problem, num_samples=5)
print(
    f"Ответ большинства: {sc['majority_answer']}  "
    f"({sc['vote_count']}/{sc['total_samples']} голосов)"
)

print("\nПрограмма мысли (сгенерированный код):")
print(program_of_thought(problem))
```

## Когда CoT помогает, а когда нет

**Помогает:**
- Многошаговая арифметика и алгебра.
- Символьное рассуждение (логические головоломки, планирование).
- Задачи, требующие объединения нескольких фактов.
- Задачи, где промежуточные шаги можно верифицировать.

**Не помогает или нейтрально:**
- Простой фактологический поиск: «Какова столица Франции?» — CoT добавляет токены и задержку без пользы.
- Задачи, где цепочка рассуждений модели может быть недостоверной — лишь ретроспективное обоснование, а не причина ответа.
- Приложения реального времени, где задержка критична и ответ краткосрочен.

**Эмерджентное поведение:** Wei et al. показали, что CoT резко улучшает результаты только выше определённого масштаба модели (примерно 100B параметров на момент публикации). С современными меньшими моделями, дообученными на данных рассуждений, порог значительно снизился.

## Варианты и расширения

**Промптинг от простого к сложному (Least-to-most):** декомпозиция сложной задачи на более простые подзадачи с последовательным решением каждой, где каждое решение используется как контекст для следующего.

**Дообучение на примерах рассуждений:** модели обучаются на датасетах с правильными трассами рассуждений (например, GSM8K, MATH), делая поведение CoT встроенным, а не промптируемым.

**Модели вознаграждения за процесс (PRM):** вместо вознаграждения за правильность финального ответа применяются пошаговые сигналы вознаграждения для направления поиска по деревьям рассуждений. Используется в o1/o3 от OpenAI.

**Достоверный CoT (Faithful CoT):** исследовательское направление, стремящееся гарантировать, что сгенерированная цепочка рассуждений причинно объясняет вывод модели, а не является ретроспективным нарративом.

**Расширенное мышление:** Claude 3.7 Sonnet предоставляет блок контента `thinking` — отдельный бюджет токенов для внутренних рассуждений, отделённый от финального ответа. Это производственная реализация CoT-блокнота с явным выделением вычислений.

## Практические применения

- **Многошаговые количественные финансы:** декомпозиция задач оценки — оценить рост выручки, применить ставку дисконтирования, получить стоимость акционерного капитала.
- **Генерация кода:** создание плана тестирования или алгоритмической схемы перед написанием кода — форма CoT, улучшающая правильность.
- **Медицинская диагностика:** дифференциальная диагностика, где модель перечисляет аргументы за и против каждой гипотезы.
- **Юридические рассуждения:** выявление релевантных прецедентов, затем обоснование заключения.
- **Обучение:** показывать ход решения пошагово, а не только финальный ответ.

## Теоретические пределы

CoT даёт трансформеру больше вычислений на этапе инференса, но не меняет, *какие* задачи в принципе решаемы. Линия аргументов Кантор-Гёдель-Тьюринг ставит жёсткий потолок:

- **[[godel-incompleteness|Теоремы Гёделя о неполноте]]** означают, что в любой достаточно выразительной формальной системе существуют истинные утверждения, которые не может вывести никакая цепочка вывода. Никакая длина CoT не пробьёт эту стену.
- **[[cantor-diagonal|Диагональный аргумент Кантора]]** — прототип любого результата вида «нельзя перечислить изнутри»; та же форма доказывает проблему остановки Тьюринга и неполноту Чайтина.
- **[[kolmogorov-complexity|Колмогоровская сложность]]** — алгоритмическая версия: у большинства двоичных строк нет описания короче их самих, и никакое сжатие — каким бы умным ни было рассуждение — не восстановит их. CoT не может «наколдовать» информацию, которой не было в промпте или весах.

Эмпирически [[mechanistic-interpretability|механистическая интерпретируемость]] изучает, какие reasoning-схемы реально живут внутри трансформера и где они ломаются. Граница между интерпретируемыми reasoning-схемами LLM и недоступными им утверждениями — современное эмпирическое лицо той же стены, которую обнаружили Гёдель и Кантор.

Именно поэтому CoT, ToT и PoT правильно понимать как **амортизацию фиксированного информационного бюджета по времени поиска**, а не как механизмы, поднимающие потолок разрешимости.

## Связанные темы

- [[speculative-decoding]] — модели с расширенным мышлением генерируют длинные цепочки рассуждений; спекулятивное декодирование ускоряет эту генерацию.
- [[rag]] — многошаговые рассуждения над найденными документами выигрывают от структурированной декомпозиции CoT.
- [[mcp]] — шаги рассуждения в агентном цикле могут инициировать вызовы инструментов MCP.
- [[godel-incompleteness]] — формальные пределы любой системы рассуждений.
- [[cantor-diagonal]] — диагональные аргументы и невычислимость.
- [[kolmogorov-complexity]] — информационно-теоретический пол того, что может восстановить рассуждение.
- [[mechanistic-interpretability]] — как reasoning-схемы выглядят внутри модели.
