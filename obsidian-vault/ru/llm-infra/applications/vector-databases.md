## Проблема: проклятие размерности

Современные эмбеддинги LLM живут в $\mathbb{R}^{1536}$ (OpenAI) или $\mathbb{R}^{4096}$ (более крупные модели). Поиск $K$ ближайших соседей методом перебора—вычисление расстояния от запроса $\mathbf{q}$ до всех $N$ векторов—стоит $O(N \cdot d)$. Для $N = 10^7$ документов и $d = 1536$ это даёт 1.5 млрд операций расстояния на запрос: неприемлемо в production latency.

**Приблизительный поиск ближайших соседей (ANN)** решает эту проблему, жертвуя малой потерей recall для достижения ускорения в 100–1000 раз, что позволяет делать семантический поиск в масштабе.

## Метрики сходства

Даны эмбеддинги $\mathbf{a}, \mathbf{b} \in \mathbb{R}^d$:

- **Косинусное сходство**: $\text{sim}(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{|\mathbf{a}| \cdot |\mathbf{b}|}$. Для единичных нормализованных векторов (стандарт в LLM workflow), косинусное сходство равно скалярному произведению и быстрее всего вычисляется на GPU.

- **L2-расстояние**: $d_2(\mathbf{a}, \mathbf{b}) = \sqrt{\sum_{i=1}^d (a_i - b_i)^2}$. Находит точных соседей, но менее естественно для эмбеддингов.

- **Скалярное произведение**: $\langle \mathbf{a}, \mathbf{b} \rangle = \mathbf{a}^T \mathbf{b}$. Идентично косинусу для нормализованных векторов; родное для GPU матрица-матрица операций.

**Соглашение**: храним единичные нормализованные эмбеддинги; вычисляем скалярное произведение для скорости.

## IVF: Инвертированный файловый индекс

**Идея**: разбить векторное пространство на $K$ кластеров через k-means, индексировать каждый кластер отдельно.

**Построение**:
1. Запустить k-means на выборке векторов, получить $K$ центроидов $\mathbf{c}_1, \ldots, \mathbf{c}_K$.
2. Для каждого вектора $\mathbf{v}$ вычислить $j^* = \arg\min_j d_2(\mathbf{v}, \mathbf{c}_j)$; хранить $\mathbf{v}$ в инвертированном списке $L_{j^*}$.

**Запрос**:
1. Найти $C$ ближайших центроидов к запросу $\mathbf{q}$ (например, $C = 10$).
2. Поискать векторы в списках $L_{j^*}$ для $C$ ближайших центроидов.
3. Вернуть top-$K$ соседей.

**Компромисс**: 
- Коэффициент сокращения: $\text{probe} = C/K$ (доля базы данных, которая просматривается).
- При $K=1000, C=10$: просмотрим только 1% векторов.
- Recall vs. скорость контролируется параметром $C$ (выше $C$ = выше recall, медленнее).
- Оптимально для больших, хорошо кластеризуемых датасетов; деградирует если кластеры перекрываются.

## HNSW: Иерархический навигируемый малый мир

**Малков & Яшунин, 2016**: многоуровневый иерархический граф для $O(\log N)$ жадного поиска.

**Архитектура**:
- Построить **многоуровневый граф**. Элемент $i$ на уровне $l$ имеет степень $M$ (адаптивный параметр).
- Верхние уровни: разреженные (сетевая магистраль), включают дальние прыжки. Нижний уровень: плотный, включает мелкозернистую навигацию.
- Рёбра каждого узла соединяют $M$ ближайших соседей на его уровне и ниже (обратные связи неявные).

**Назначение уровней**:
$$P(\text{элемент на уровне} \geq l) = e^{-l / \ln(M)}$$

Типичное $M=5$: средняя высота $\sim \log_2(N)$.

**Построение**:
1. Назначить новому элементу случайную высоту уровня $h$.
2. Вставить на уровни $0$ до $h$:
   - На верхнем уровне найти ближайший жадным поиском от точки входа; добавить двусторонние рёбра к $M$ ближайшим.
   - Перейти на уровень ниже, повторить с расширенным набором кандидатов.

**Поиск запроса**:
1. Начать с точки входа на верхнем уровне.
2. Жадный луч-поиск с шириной $ef$ (ширина поиска) кандидатов:
   - Расширить границу к ближайшим соседям, обрезать до $ef$ лучших.
   - Спуститься на следующий уровень; повторить.
3. Вернуть top-$K$ с уровня 0.

**Сложность**: 
- Построение: $O(N \cdot M)$ для $M = 5, 10$ (низкая константа).
- Запрос: $O(\log N)$ ожидаемое время; обычно $10$–$50$ вычислений расстояния для миллиардных масштабов (против $10^6$ для IVF на том же масштабе).

**Преимущества**: 
- Асимптотически быстрейший для высокого recall ANN в высоких измерениях.
- Без кластеризации или предварительной обработки; инкрементальная вставка естественна.
- Один параметр ($M$) настраивает память vs. скорость.

## Product Quantization (PQ)

**Проблема**: хранение $10^7 \times 1536$ float'ов = 60GB памяти при одинарной точности.

**Идея**: расщепить вектор на $M$ неперекрывающихся подвекторов, квантизировать каждый независимо.

$$\mathbf{v} = [\mathbf{v}^{(1)}, \ldots, \mathbf{v}^{(M)}], \quad \mathbf{v}^{(j)} \in \mathbb{R}^{d/M}$$

Для каждого подпространства $j$ запустить k-means с $K$ центроидами; кодировать каждый подвектор как один байт (индекс $0$ до $K-1$, $K \le 256$).

**Хранение**: $M$ байт на вектор против $4d$ байт (32-bit float), $150\times$ сжатие. Вычисление расстояния использует таблицы поиска, избегая распаковки.

**Потеря реконструкции**: 4–8% падение recall против точных соседей; приемлемо для top-100 retrieval в retrieval-augmented generation (RAG).

## FAISS: Стандартная библиотека

Facebook AI Similarity Search библиотека:

- **IndexFlatL2**: точный brute-force; базовый.
- **IndexIVFFlat**: IVF + точное расстояние в кластерах; $10$–$100\times$ ускорение, настраиваемый recall через `nprobe`.
- **IndexIVFPQ**: IVF + product quantization; миллиардные масштабы, GPU ускорение.
- **GPU поддержка**: batch query в $10$–$100\times$ раз быстрее чем CPU.

Типичная миллиардная конфигурация: 100k кластеров, $C=100$ проб, PQ с $M=96, K=256$. Latency запроса: 10–50ms; RAM на вектор: 1 байт.

## Фильтрованный поиск

На практике: найти документы по эмбеддингу **и** метаданным (дата, категория, пользователь).

**Компромиссы**:
- **Pre-filter**: применить WHERE условие, затем ANN на отфильтрованном наборе. Быстро если мало документов match; медленно если selectivity низкая (высокий recall).
- **Post-filter**: ANN возвращает $K' \gg K$, отфильтровать после. Тратит вычисления; требуется если filter matches мало элементов.
- **Hybrid**: candidate generation (ANN) + reranking (filters + dense retriever).

При selectivity $< 1\%$ (например, документы за последние 7 дней), pre-filter + ANN доминирует. Qdrant и Milvus поддерживают pre-filtering внутри ANN loop для лучшего результата обоих.

## Production системы

| Система | Хранение | GPU | Фильтры | Масштаб |
|---------|----------|-----|---------|---------|
| **Pinecone** | Managed | ✓ | ✓ | 100B+ векторов |
| **Qdrant** | Self-hosted | ✓ | ✓✓ | 100M–1B |
| **Weaviate** | Self-hosted | ✓ | ✓ | 100M–1B |
| **Chroma** | In-process | ✗ | ✓ | 10M |
| **pgvector** | PostgreSQL | ✗ | ✓✓ | 100M (IVF) |

**Managed vs. Self-hosted**:
- Pinecone: serverless, автоматическое масштабирование, HNSW + PQ, дорого (~$0.40/M векторов/месяц).
- Qdrant/Weaviate: open-source, persistent, полный контроль, управляй сам.
- pgvector: используй существующий Postgres; медленнее но тесная интеграция, native ACID.

## Роль в RAG

[[rag|RAG]] pipeline:

1. **Индексирование**: разбить документы на chunks (512–1024 токенов), embedded каждый с [[embedding-models]].
2. **Хранение**: векторы → vector DB (с chunk ID, метаданные).
3. **Запрос**: embedded пользовательский вопрос с той же моделью.
4. **Retrieval**: top-$K$ ANN (например, $K=10$) из vector DB.
5. **Контекст**: retrieve chunk text, передать в LLM в system prompt или context window.

[[llm|LLM]] augmentation через vector retrieval достигает:
- Сниженная hallucination (grounded в retrieved facts).
- Актуальное знание (re-index новые документы, no model retraining).
- Long-context handling (retrieve только relevant chunks, не весь corpus).
